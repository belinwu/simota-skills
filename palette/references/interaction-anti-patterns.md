# Interaction Anti-Patterns

> UIインタラクション設計の落とし穴、コンポーネント操作の問題パターン、防止策

## 1. インタラクション 10 大アンチパターン

| # | アンチパターン | 問題 | ユーザーへの影響 | 対策 |
|---|-------------|------|---------------|------|
| **IA-01** | **Hide and Hover** | アクション（編集/削除）がホバー時のみ表示 | 操作可能な要素が発見不能、タッチデバイスで完全に機能しない | 常時表示 or ケバブメニュー（⋮）で一貫提供 |
| **IA-02** | **小さなクリックターゲット** | リンク/ボタンの対象領域が小さすぎる | Fitts' Law 違反、精密操作が必要で疲労増加 | 最小 44×44px（WCAG 2.2 AA: 24×24px） |
| **IA-03** | **過剰エンジニアリング** | 単純な操作に D&D やフェードエフェクトを適用 | 学習コスト増加、操作効率低下 | KISS 原則、既存メンタルモデルに合致する操作 |
| **IA-04** | **バッチ操作の欠如** | リスト項目を 1 つずつしか処理できない | 大量操作で時間浪費、フラストレーション | 複数選択 + 一括アクション（削除/アーカイブ等） |
| **IA-05** | **曖昧なリンクラベル** | 「ここをクリック」「詳細」など | スクリーンリーダーで文脈不明、操作結果が予測不能 | 動詞 + 対象（「プロジェクトを作成」） |
| **IA-06** | **ラベルなしの入力要素** | Radio/Checkbox に `<label>` 未関連付け | クリック領域が極小、a11y 非対応 | `<label htmlFor>` で要素と関連付け |
| **IA-07** | **ハッピーパス限定設計** | 正常系のみ設計、エラー/空/ローディング未考慮 | エラー時にユーザーが行き詰まる | 全状態の設計必須（Empty/Error/Loading/Offline） |
| **IA-08** | **UI 肥大化** | 機能を詰め込みすぎたインターフェース | 選択肢過多で決断麻痺、本質機能の埋没 | 80/20 ルール、段階的開示 |
| **IA-09** | **サイレントフェイル** | エラー発生時に console.error のみで UI 無反応 | ユーザーが操作結果を認知できない | 全操作にフィードバック（Toast/Inline alert） |
| **IA-10** | **確認なし破壊操作** | 削除/リセット/解除が即実行 | 誤操作でデータ喪失、心理的不安 | 確認ダイアログ or Soft Delete + Undo |

## 2. コンポーネント固有のアンチパターン

```
モーダル:
  ❌ ページロード時に即座にモーダル表示（ユーザーの文脈を無視）
  ❌ モーダル内にモーダル（ネスト）
  ❌ フォーカストラップの未実装（背面クリック可能）
  ❌ ESC キーで閉じられない
  ✅ ユーザーアクションでのみ表示
  ✅ 単一モーダル原則
  ✅ フォーカストラップ + ESC + 背面クリックで閉じる

ツールチップ:
  ❌ 重要な操作情報をツールチップ内に隠す
  ❌ タッチデバイスで表示不可能な設計
  ❌ ツールチップ内にインタラクティブ要素（リンク/ボタン）
  ✅ 補足情報のみ、必須情報は常時表示
  ✅ 代替手段（aria-describedby）を併用

カルーセル:
  ❌ 自動回転（ユーザーの読速に非同期）
  ❌ スワイプ/矢印以外の操作手段なし
  ❌ 現在位置の不明瞭なインジケータ
  ✅ ユーザー制御のみ（手動操作）
  ✅ 全スライドのリスト表示オプション
  ✅ prefers-reduced-motion 時は自動回転停止

ドロップダウン:
  ❌ 7+ 項目で検索フィルターなし
  ❌ キーボード操作（Arrow/Enter/Escape）非対応
  ❌ 選択肢の頻度/重要度を無視した並び順
  ✅ 5 項目超は検索可能 Combobox
  ✅ 最頻選択肢を上位に配置
```

## 3. フィードバック設計の原則

```
フィードバック応答時間ガイドライン（Jakob Nielsen）:

  0-100ms:   即時応答。ユーザーは遅延を感じない
  100-300ms: 軽い遅延。視覚的フィードバック推奨
  300ms-1s:  明確な遅延。ローディングインジケータ必須
  1s-10s:    長い待機。プログレスバー + キャンセル可能
  10s+:      バックグラウンド処理 + 通知で完了報告

フィードバックの 4 層:

  Layer 1: Immediate（即時）
    - ボタンのホバー/プレス状態変化
    - フォーカスリングの表示
    - タップフィードバック

  Layer 2: Acknowledgment（確認応答）
    - ローディングスピナー/スケルトン
    - 「処理中...」表示
    - プログレスバー

  Layer 3: Result（結果）
    - 成功 Toast（3 秒自動消去）
    - エラーメッセージ（手動消去 or 5 秒）
    - インラインバリデーション結果

  Layer 4: Recovery（回復）
    - Undo アクション（5 秒以内）
    - リトライボタン
    - エラーからの回復ガイダンス
```

## 4. Palette との連携

```
Palette での活用:
  1. OBSERVE フェーズで IA-01〜10 のスクリーニング
  2. MICRO レンズでコンポーネント固有アンチパターン検出
  3. MESO レンズでフィードバック層の完全性チェック
  4. IMPLEMENT フェーズでフィードバック 4 層の適用

品質ゲート:
  - ホバーのみ表示のアクション → 常時表示 or メニュー化（IA-01 防止）
  - クリックターゲット < 44px → サイズ拡大を要求（IA-02 防止）
  - 非同期操作のローディング状態なし → フィードバック追加（IA-09 防止）
  - 破壊操作の確認なし → 確認ダイアログ or Undo 追加（IA-10 防止）
  - モーダルの ESC/フォーカストラップ未実装 → 修正必須
```
