# DAST Anti-Patterns & False Positive Management

> DAST 実施時の落とし穴、偽陽性管理、トリアージ戦略、Proof-Based Scanning

## 1. DAST 8 大アンチパターン

| # | アンチパターン | 問題 | 対策 |
|---|-------------|------|------|
| **DA-01** | **偽陽性の放置** | アラート疲労 → 開発者が全アラートを無視 → 本物の脆弱性を見逃す | Proof-Based Scanning 導入 · 偽陽性の体系的トリアージプロセス確立 |
| **DA-02** | **一律スキャン** | 全コミットで同一フルスキャン → パイプライン遅延 · リソース浪費 | 差分スキャン（変更箇所のみ）· フルスキャンは夜間/週次に |
| **DA-03** | **認証なしスキャン** | 保護された領域をテストできない → 浅いカバレッジ · 不正確な結果 | 認証情報の安全な管理 · セッション処理の適切な設定 |
| **DA-04** | **ツールデフォルト依存** | 工場出荷設定のまま → フレームワーク固有の安全なパターンを脆弱性と誤検出 | アプリ固有のルールチューニング · フレームワーク認識設定 |
| **DA-05** | **自動スキャンのみ** | ビジネスロジック脆弱性を検出不能 → 表面的なカバレッジ | 自動 + 手動テストの組み合わせ · ビジネスフロー固有のテストケース作成 |
| **DA-06** | **本番直接スキャン** | データ改変リスク · サービス中断 · 法的問題 | Staging/Pre-production 環境でテスト · 本番は受動スキャンのみ |
| **DA-07** | **コンテキスト無視** | 全アラートを同一重要度で扱う → 優先順位の混乱 | CVSS + 資産価値 + 悪用可能性の複合評価 |
| **DA-08** | **フィードバック不在** | 検出結果を開発者に返さない → 同じ脆弱性の再発 | 自動チケット連携 · PR コメント · 修正ガイダンス付き通知 |

## 2. 偽陽性の根本原因

### 3 大原因

```
原因 1: 過度なパターンマッチング
  - 文字列一致のみで判断 → コンテキスト不在
  - 例: 入力が反映されただけで XSS と判定
  - 実際にはフレームワークの自動エスケープで安全

原因 2: アプリケーションコンテキスト不足
  - レガシースキャナーはアプリ設定・ビジネスロジック・カスタムセキュリティ制御を理解しない
  - 不完全なデータに基づく誤った判断
  - WAF のブロック応答を脆弱性と誤認

原因 3: 認証・セッション処理の失敗
  - 動的ワークフローのナビゲーション不能
  - 保護領域にアクセスできず → 浅い非認証スキャン
  - 部分的・非アクション可能・不正確な結果を生成
```

### 偽陽性発生パターン

| パターン | 例 | 検出困難度 |
|---------|---|----------|
| **WAF 誤認** | WAF がブロックした応答を脆弱性と判定 | 低（設定で回避可） |
| **フレームワーク安全機能** | Spring Security / テンプレートエスケープを未考慮 | 中（チューニング必要） |
| **ビジネスロジック誤解** | 正常な機能動作を異常と判定 | 高（手動レビュー必要） |
| **動的コンテンツ誤認** | SPA の動的レンダリングを脆弱性と判定 | 中（Ajax Spider 設定） |

## 3. Proof-Based Scanning

### 原則

```
従来の DAST: 疑わしいパターンを検出 → 報告（推測ベース）
Proof-Based: 安全に悪用を試行 → 成功した場合のみ報告（証拠ベース）

利点:
  - 偽陽性の大幅削減（報告前に悪用可能性を確認）
  - 各アラートに request/response ペアの証拠付き
  - 開発者の信頼を維持
  - トリアージ工数の削減

対象脆弱性クラス:
  - SQL Injection: 安全なクエリで確認（SLEEP, BENCHMARK）
  - XSS: DOM 操作で確認（alert ではなくカスタムマーカー挿入）
  - Command Injection: 無害なコマンドで確認（echo マーカー）
  - Path Traversal: 既知ファイル読み取りで確認（/etc/hostname）
```

## 4. トリアージ戦略

### 4 段階トリアージモデル

| 段階 | 手法 | 精度 | コスト |
|------|------|------|------|
| **L1** | 自動フィルタリング（既知の偽陽性パターン除外） | 中 | 極低 |
| **L2** | クロスツール相関（SAST + DAST + IAST の一致） | 高 | 低 |
| **L3** | ML ベース分類（過去のトリアージ結果から学習） | 高 | 中 |
| **L4** | 手動検証（セキュリティエンジニアによる確認） | 最高 | 高 |

### ML ベーストリアージ

```
Rapid7 研究結果:
  - ハイブリッド DAST/ML モデルで偽陽性の 96% をフィルタリング
  - 過去のトリアージ結果で適応モデルをトレーニング
  - 反復的な無害パターンを自動抑制
  - 人間のレビュー + 監査ログとの併用が必須（安全性と説明可能性の確保）

実装アプローチ:
  1. 過去 6 ヶ月のトリアージデータを収集
  2. 特徴量: スキャンルール ID、URL パターン、レスポンスコード、パラメータ名
  3. 分類モデル: True Positive / False Positive / Needs Review
  4. 信頼度閾値: 90% 以上で自動分類、未満は L4 手動レビュー
```

### トリアージ判断フロー

```
スキャン結果受信
  │
  ├─ 既知の偽陽性パターンか？
  │   Yes → L1: 自動除外（ログ記録）
  │   No ↓
  │
  ├─ 他のツール（SAST/IAST）でも検出されたか？
  │   Yes → 優先度引き上げ → L4 即時レビュー
  │   No ↓
  │
  ├─ Proof-Based 証拠があるか？
  │   Yes → True Positive として確定 → チケット作成
  │   No ↓
  │
  ├─ ML モデルの分類結果は？
  │   FP 信頼度 > 90% → L3: 自動抑制（監査ログ付き）
  │   TP 信頼度 > 90% → チケット作成
  │   それ以外 → L4: 手動レビュー
  │
  └─ 手動レビュー結果をモデルにフィードバック
```

## 5. Probe との連携

```
Probe での活用:
  1. Scan フェーズで DA-01〜08 のチェックを適用
  2. 偽陽性トリアージに 4 段階モデルを使用
  3. Proof-Based Scanning を優先的に採用
  4. 検出結果の自動チケット連携を推奨

品質ゲート:
  - 認証なしスキャンのみ → 警告（DA-03 防止）
  - ツールデフォルト設定 → チューニング提案（DA-04 防止）
  - 自動スキャンのみ → 手動テスト追加を推奨（DA-05 防止）
  - 偽陽性率 > 30% → ルールチューニングを要求（DA-01 防止）
  - フルスキャンがパイプラインで 30 分超 → 差分スキャンへ切り替え提案（DA-02 防止）
```
