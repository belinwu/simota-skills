# Organizational Complexity Anti-Patterns

> プロセス肥大化、会議過多、官僚主義の検出と削減、アジャイルセレモニーの蓄積、意思決定フレームワーク

## 1. 組織的複雑性 8 大アンチパターン

| # | アンチパターン | 症状 | Void の問い |
|---|-------------|------|-----------|
| **OC-01** | **承認レイヤー過多** | 小さな変更に 3+ の承認が必要 → 意思決定の遅延 | この承認は何を防いでいるか？実際に防いだか？ |
| **OC-02** | **会議肥大化** | 「念のため」参加者の増加 → 決定不能な大人数会議 | この会議の意思決定者は誰か？残りは通知で十分か？ |
| **OC-03** | **プロセスの化石化** | 理由を忘れたまま続くプロセス → 形骸化した儀式 | このプロセスを始めた理由は今も有効か？ |
| **OC-04** | **ドキュメント増殖** | 更新されない仕様書が蓄積 → 矛盾する複数の「真実」 | このドキュメントは最後にいつ参照されたか？ |
| **OC-05** | **ツール乱立** | 同種ツールが複数存在 → 情報の分散・学習コスト | 1 つのツールに統合できないか？ |
| **OC-06** | **セレモニー蓄積** | アジャイル儀式が積み重なり → 作業時間の圧迫 | この儀式は今のチームサイズで必要か？ |
| **OC-07** | **コミッティー増殖** | あらゆる判断に委員会 → 責任の拡散・速度低下 | 1 人の判断で十分な範囲ではないか？ |
| **OC-08** | **報告義務の積層** | 週報・月報・四半期レポートが重複 → 作成工数 > 価値 | このレポートの読者は誰か？実際に読まれているか？ |

## 2. プロセス肥大化の検出

### プロセスの CoK 評価

| 次元 | 0（低コスト） | 10（高コスト） | 計測方法 |
|------|-------------|---------------|---------|
| **Upkeep** | 自動化済み・手間なし | 毎回手動で多工数 | 実行にかかる人時/回 |
| **Verification** | 結果が明確・検証不要 | 成功/失敗の判定が曖昧 | 品質チェックの工数 |
| **Cognitive Load** | 直感的・説明不要 | 暗黙知が必要・例外多数 | 新メンバーの習得時間 |
| **Entanglement** | 独立・他に影響なし | 他プロセスと密結合 | 変更時の連鎖影響数 |
| **Replaceability** | 代替手段あり | 代替不能 | 廃止時の影響範囲 |

### プロセス健全性の指標

| 指標 | 健全 | 要注意 | 危険 |
|------|------|--------|------|
| **作業時間に占める会議率** | < 20% | 20-40% | > 40% |
| **意思決定までのリードタイム** | < 2 日 | 2-7 日 | > 7 日 |
| **承認レイヤー数** | 1-2 | 3 | 4+ |
| **新メンバーのプロセス習得時間** | < 1 週間 | 1-4 週間 | > 1 ヶ月 |
| **形骸化したプロセス率** | < 10% | 10-30% | > 30% |

## 3. Amazon「官僚主義メールボックス」モデル

### 2024 年の取り組み

```
背景:
  - Amazon が全社員向けに「官僚主義メールボックス」を設置（2024年）
  - CEO Andy Jassy: "We want to operate like the world's largest startup"
  - 目的: 不要なプロセス・承認・ルールを従業員から直接報告させる

仕組み:
  1. 従業員が不要と感じるプロセスをメールで報告
  2. 専任チームが各報告を調査
  3. 不要と判断されたプロセスを廃止
  4. 結果をフィードバック

Void への適用:
  - プロセス監査の「ボトムアップ」アプローチとして参考
  - 現場の声 = プロセスの実態データ
  - 報告パターンの集約 → 組織的アンチパターンの特定
```

## 4. アジャイルセレモニーの蓄積問題

### セレモニー CoK 評価

| セレモニー | 目的 | CoK が高くなる条件 | 軽量化の選択肢 |
|-----------|------|-------------------|--------------|
| **デイリースタンドアップ** | 同期・障害共有 | チームが小さい/非同期でも十分 | 非同期テキスト更新（Slack） |
| **スプリントプランニング** | 作業計画 | バックログが整備済み | 軽量版（30 分以内） |
| **レトロスペクティブ** | 改善 | 形骸化・同じ話題の繰り返し | 隔週化 or 非同期 |
| **スプリントレビュー** | デモ・フィードバック | ステークホルダーが来ない | 録画共有 + 非同期フィードバック |
| **バックログリファインメント** | 準備 | 過剰な事前分析 | JIT リファインメント |

### セレモニー監査フロー

```
各セレモニーについて:
  │
  ├─ 参加者は全員が発言/貢献しているか？
  │   No → 参加者を削減 or オプション化
  │   Yes ↓
  │
  ├─ 同等の成果を非同期で得られるか？
  │   Yes → 非同期化を検討
  │   No ↓
  │
  ├─ 頻度は適切か？（毎回新しい価値が生まれているか？）
  │   No → 頻度を下げる
  │   Yes ↓
  │
  ├─ 時間は適切か？（制限時間内に終わっているか？）
  │   No → 時間を短縮 or アジェンダを絞る
  │   Yes ↓
  │
  └─ 維持（現状の形式で価値あり）
```

## 5. 意思決定の複雑性削減

### 意思決定フレームワーク

| フレームワーク | 適用場面 | 特徴 |
|-------------|---------|------|
| **RACI** | 責任分担の明確化 | R(実行)A(承認)C(相談)I(報告) |
| **DACI** | 意思決定の迅速化 | D(推進)A(承認)C(貢献)I(情報) |
| **One-way/Two-way Door** | 判断速度の最適化 | 可逆 = 即決、不可逆 = 慎重 |

### Amazon の "Two-way Door" 原則

```
One-way Door（不可逆）:
  - 慎重な分析・合議が必要
  - 例: 大規模アーキテクチャ変更、プロダクト廃止、契約

Two-way Door（可逆）:
  - 素早く決定・実行・修正
  - 例: UI 変更、A/B テスト、設定変更、プロセス試行
  - 問題: 多くの組織が Two-way Door を One-way Door として扱う
    → 不要な承認レイヤー → 速度低下

Void の適用:
  - 承認プロセスが Two-way Door に適用されている場合 → 簡素化提案
  - 「このプロセスで守っているものは可逆か不可逆か？」を問う
```

## 6. ドキュメント・報告の削減

### ドキュメント監査基準

| 基準 | 検出条件 | アクション |
|------|---------|----------|
| **未参照** | 6 ヶ月以上閲覧なし | REMOVE 候補 |
| **未更新** | 12 ヶ月以上更新なし | 内容確認 → REMOVE or UPDATE |
| **重複** | 同一内容が 2 箇所以上 | 統合（Single Source of Truth） |
| **矛盾** | 複数ドキュメントで異なる記述 | 即時解決（最新を正とする） |
| **過剰詳細** | 読者に不要な詳細が大半 | 簡素化 or 階層化 |

### レポート削減の意思決定

```
各レポートについて:
  │
  ├─ 読者は特定できるか？
  │   No → REMOVE（誰も読んでいない）
  │   Yes ↓
  │
  ├─ 読者はアクションを取っているか？
  │   No → 形式変更 or REMOVE
  │   Yes ↓
  │
  ├─ 他のレポートと内容が重複しているか？
  │   Yes → 統合
  │   No ↓
  │
  ├─ 自動生成できるか？
  │   Yes → 自動化（手動作成を廃止）
  │   No ↓
  │
  └─ 維持（ただし頻度を見直し）
```

## 7. Void との連携

```
Void での活用:
  1. QUESTION フェーズで OC-01〜08 のパターンをプロセス/ドキュメント対象に適用
  2. WEIGH フェーズでプロセス版 CoK Score を使用（作業時間/会議率/承認数で計測）
  3. SUBTRACT フェーズで Process Pruning / Document Retirement パターンを適用
  4. PROPOSE フェーズで Two-way Door 原則を判断根拠に

品質ゲート:
  - 会議率 > 40% → OC-02 セレモニー監査を提案
  - 承認レイヤー > 3 → OC-01 簡素化を提案
  - 未参照ドキュメント > 30% → OC-04 一括廃止を提案
  - 新メンバーのプロセス習得 > 1 ヶ月 → OC-03 プロセス簡素化を提案
  - 同種ツール > 2 → OC-05 統合を提案
```
