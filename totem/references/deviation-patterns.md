逸脱の重要度分類、false positive回避ルール、文化ドリフト検出、意図的逸脱判別。

## Deviation Severity Classification

### Severity Matrix

| Severity | Criteria | Example | Action |
|----------|----------|---------|--------|
| **HIGH** | 2+ 次元違反、public interface に影響、確立パターンを破壊 | camelCase コードベースの public API に snake_case 関数 | → Judge にレビューブロック推奨として通知 |
| **MEDIUM** | 1次元の有意な違反、内部コード | 新モジュールで異なるエラー処理パターン | → Deviation Report に記録 |
| **LOW** | 軽微な不統一、精神は準拠しているが字面が異なる | コメントスタイルの微差 | → INFO としてログ |
| **INFO** | 検出されたが意図的または無関係の可能性が高い | テストファイルが異なる命名を使用（想定内） | → アクション不要 |

### Severity Decision Tree

```
逸脱を検出
  │
  ├─ Public interface に影響? ──YES──→ HIGH 候補
  │    │
  │    └─ 2+ 次元に違反? ──YES──→ HIGH 確定
  │         │
  │         └─ NO → MEDIUM
  │
  ├─ 1次元に有意な違反? ──YES──→ MEDIUM 候補
  │    │
  │    └─ 内部コードのみ? ──YES──→ MEDIUM 確定
  │         │
  │         └─ NO → HIGH に昇格
  │
  ├─ 精神は準拠? ──YES──→ LOW
  │
  └─ サブカルチャー内? ──YES──→ INFO
```

### Severity Score Calculation

```yaml
severity_scoring:
  base_score: "違反次元数 × 2"
  modifiers:
    public_interface: "+3"
    multiple_dimensions: "+2 per additional dimension"
    new_code_by_existing_contributor: "+1"
    new_code_by_new_contributor: "+0 (learning curve expected)"
    high_traffic_file: "+2"
    legacy_file: "-2"
  thresholds:
    HIGH: ">= 5"
    MEDIUM: "3-4"
    LOW: "1-2"
    INFO: "0"
```

## False Positive Exclusion Rules

### Rule 1: Generated Code

```yaml
rule: "generated_code_exclusion"
detection:
  - file_header: "// Code generated by .* DO NOT EDIT"
  - file_header: "# Auto-generated"
  - file_header: "/* eslint-disable */"
  - file_path: "**/generated/**"
  - file_path: "**/__generated__/**"
  - gitattributes: "linguist-generated=true"
action: "完全除外 — DNA Profile に含めない"
```

### Rule 2: Vendored / Third-Party Code

```yaml
rule: "vendor_exclusion"
detection:
  - file_path: "**/vendor/**"
  - file_path: "**/third_party/**"
  - file_path: "**/node_modules/**"
  - file_path: "**/external/**"
action: "完全除外"
```

### Rule 3: Module Subcultures

```yaml
rule: "subculture_exclusion"
detection:
  - file_path: "**/test/**"
  - file_path: "**/__tests__/**"
  - file_path: "**/*_test.*"
  - file_path: "**/scripts/**"
  - file_path: "**/tools/**"
  - file_path: "**/migrations/**"
action: "メインスコアから除外 → Module Subcultures セクションに記録"
```

### Rule 4: Legacy Code Marked for Deprecation

```yaml
rule: "deprecated_code_exclusion"
detection:
  - annotation: "@deprecated"
  - comment: "// DEPRECATED"
  - file_header: "# This module is scheduled for removal"
  - directory_marker: "DEPRECATED.md in directory"
action: "逸脱として報告しない（既知の技術的負債）"
```

### Rule 5: Framework-Mandated Patterns

```yaml
rule: "framework_mandated_exclusion"
examples:
  - pattern: "React hooks must start with 'use'"
    framework: "React"
    dimension: "N (Naming)"
  - pattern: "Go exported names must be PascalCase"
    framework: "Go"
    dimension: "N (Naming)"
  - pattern: "Django models inherit from models.Model"
    framework: "Django"
    dimension: "A (Abstraction)"
  - pattern: "Jest expects describe/it blocks"
    framework: "Jest"
    dimension: "T (Testing)"
action: "フレームワーク要件として逸脱扱いしない"
```

### Rule 6: Configuration Files

```yaml
rule: "config_file_exclusion"
detection:
  - extension: ".json, .yaml, .yml, .toml, .ini, .cfg, .env"
  - file_path: "**/.*rc"
  - file_path: "**/*.config.*"
action: "Naming 次元のスコアリングから除外（固有の規約）"
note: "D (Dependencies) の分析ソースとしては使用"
```

### Rule 7: One-Off Scripts and Migrations

```yaml
rule: "one_off_exclusion"
detection:
  - file_path: "**/scripts/**"
  - file_path: "**/migrations/**"
  - file_path: "**/seeds/**"
  - shebang: "#!/usr/bin/env"
action: "メインスコアへの影響を 0.5x に減衰"
```

## Intentional Deviation Detection

意図的な逸脱かどうかを判別するための調査手順。

### Investigation Checklist

```yaml
intentional_deviation_checks:
  - check: "git blame 調査"
    question: "誰がいつ書いたコードか?"
    indicators:
      intentional: "シニアコントリビュータによる最近のコード"
      accidental: "新しいコントリビュータによる初コミット"

  - check: "PR/MR レビュー確認"
    question: "この逸脱はレビューで承認されたか?"
    indicators:
      intentional: "レビュー付きでマージされた"
      accidental: "レビューなし or セルフマージ"

  - check: "コメント・ドキュメント確認"
    question: "逸脱を説明するコメント/ADR があるか?"
    indicators:
      intentional: "// Intentionally using X because Y"
      accidental: "説明なし"

  - check: "パターンの一貫性"
    question: "同じ逸脱が複数箇所で見られるか?"
    indicators:
      intentional: "同じパターンが一貫して使用されている（新しい規約の可能性）"
      accidental: "散発的・単発的な出現"
```

### Intentionality Score

```yaml
intentionality_scoring:
  factors:
    reviewed_and_merged: "+2"
    documented_with_comment: "+2"
    consistent_pattern: "+2"
    senior_contributor: "+1"
    recent_code: "+1"
    no_review: "-1"
    single_occurrence: "-1"
    new_contributor: "-1"
  interpretation:
    ">= 4": "LIKELY_INTENTIONAL → 規約進化の可能性"
    "1-3": "UNCERTAIN → ON_INTENTIONAL_DEVIATION トリガー"
    "<= 0": "LIKELY_ACCIDENTAL → 逸脱レポートに記録"
```

## Cultural Drift Detection

時間経過に伴う文化的変化を検出する仕組み。

### Drift Detection Method

```yaml
drift_detection:
  method: "ベースラインとの比較"
  steps:
    - step: "ベースライン DNA Profile を設定"
      action: "初回プロファイリング結果を保存"
    - step: "定期的に再プロファイリング"
      frequency: "月次 or 四半期"
    - step: "次元スコアの変化を計算"
      formula: "delta = current_score - baseline_score"
    - step: "変化の方向を分類"
      categories:
        evolution: "スコアが改善方向に変化"
        erosion: "スコアが悪化方向に変化"
        shift: "方向は中立だがパターンが変化"
```

### Drift Alert Thresholds

| Delta | Classification | Action |
|-------|---------------|--------|
| < 0.3 | 安定 | アクション不要 |
| 0.3 - 0.5 | 微小変化 | モニタリング継続 |
| 0.5 - 1.0 | 有意な変化 | ON_CULTURAL_DRIFT トリガー発火 |
| > 1.0 | 大幅な変化 | 即時レポート + ユーザーへの通知 |

### Evolution vs Erosion

```yaml
classification:
  evolution:
    definition: "スコアが改善方向に変化（0→3 方向）"
    indicators:
      - "新しいコードがより高いスコアを示す"
      - "規約が明文化された（linter rule 追加等）"
      - "チーム合意の結果として変化"
    action: "ベースラインを更新"

  erosion:
    definition: "スコアが悪化方向に変化（3→0 方向）"
    indicators:
      - "新しいコードが既存規約に従わない"
      - "linter rule の無効化・例外が増加"
      - "コントリビュータ増加に伴う規律低下"
    action: "ON_CULTURAL_DRIFT トリガー + 対策提案"

  shift:
    definition: "パターンが変化したがスコアは同等"
    indicators:
      - "camelCase → snake_case に移行（言語移行等）"
      - "テストフレームワーク変更"
    action: "ベースラインを更新 + 理由を記録"
```

### Temporal Bucketing

```yaml
temporal_bucketing:
  purpose: "時期ごとのスコア変化を可視化"
  buckets:
    - "T-3: 9-12ヶ月前"
    - "T-2: 6-9ヶ月前"
    - "T-1: 3-6ヶ月前"
    - "T-0: 直近3ヶ月"
  output: |
    | Dimension | T-3 | T-2 | T-1 | T-0 | Trend |
    |-----------|-----|-----|-----|-----|-------|
    | N: Naming | 2.0 | 2.1 | 2.3 | 2.5 | ↑ Evolution |
    | E: Errors | 1.5 | 1.5 | 1.3 | 1.0 | ↓ Erosion |
```

## Deviation Lifecycle

### State Machine

```
DETECTED → CLASSIFIED → REPORTED → ┬→ RESOLVED
                                    ├→ ACCEPTED_AS_EVOLUTION
                                    └→ FALSE_POSITIVE
```

### State Descriptions

| State | Description | Trigger |
|-------|-------------|---------|
| **DETECTED** | パターン分析で逸脱を検出 | 自動検出 |
| **CLASSIFIED** | 重要度（HIGH/MEDIUM/LOW/INFO）を割り当て | Severity Decision Tree |
| **REPORTED** | Deviation Report に記録、該当エージェントに通知 | 分類完了 |
| **RESOLVED** | 逸脱が修正された | コード修正 |
| **ACCEPTED_AS_EVOLUTION** | 意図的な規約進化と認定 | ON_INTENTIONAL_DEVIATION → ユーザー承認 |
| **FALSE_POSITIVE** | 誤検出と認定 | False Positive Rules に該当 |

### Deviation Tracking

```yaml
deviation_record:
  id: "DEV-XXXX"
  detected_at: "YYYY-MM-DD"
  dimensions: ["N", "E"]
  severity: "HIGH | MEDIUM | LOW | INFO"
  file_path: "string"
  line_range: "L10-L25"
  description: "string"
  expected_pattern: "string"
  actual_pattern: "string"
  intentionality: "INTENTIONAL | UNCERTAIN | ACCIDENTAL"
  status: "DETECTED | CLASSIFIED | REPORTED | RESOLVED | ACCEPTED | FALSE_POSITIVE"
  resolution: "string (if resolved)"
```
