# Source Preparation Guide

NotebookLMのソース資料を最適化するための詳細ガイド。

---

## Core Principle

> ソース品質が出力品質の70%を決定する。
> どれだけ優れたステアリングプロンプトも、低品質なソースを補えない。

---

## Source Type Optimization

### PDF

**強み:** 構造化された情報、図表の保持、ページ参照が可能

**最適化ポイント:**
- テキスト選択可能なPDFを使用する（スキャン画像PDFはOCR精度に依存）
- スキャンPDFの場合はOCR処理済みのものを使用
- 不要なヘッダー/フッター/ページ番号はノイズになりうる
- 目次がある場合はそのまま残す（構造理解に役立つ）
- 大きなPDFは関連セクションだけを抽出した方が品質向上

**注意点:**
- パスワード保護されたPDFはアップロード不可
- 埋め込みフォントの問題でテキスト抽出が失敗する場合がある
- 図表内のテキストは認識されない場合がある

**品質チェック:**
```
□ テキスト選択が可能か
□ 文字化けがないか
□ 図表のキャプションが読み取れるか
□ 50ページ以上の場合、関連部分を抽出したか
```

---

### Google Docs

**強み:** 見出し構造が直接活用される、リアルタイム編集可能

**最適化ポイント:**
- 見出しスタイル（H1/H2/H3）を正しく使用する — NotebookLMは構造を認識する
- コメントと提案モードの変更は削除してからリンクする
- 「表示」権限を確認（NotebookLMがアクセスできる必要がある）
- 書式設定（太字・斜体）は保持される — 重要ポイントの強調に有効

**注意点:**
- 非常に長いドキュメント（100ページ超）はパフォーマンスに影響
- 埋め込み画像のテキストは認識されない場合がある
- リンク先のコンテンツは含まれない

**品質チェック:**
```
□ 見出しスタイルを正しく適用したか
□ コメント・提案を解決/削除したか
□ 共有設定でアクセス可能か
□ 不要な書式（蛍光色ハイライト等）を整理したか
```

---

### Web URL

**強み:** 最新情報を直接参照可能、リンク共有が容易

**最適化ポイント:**
- 静的なページを優先する（動的にロードされるコンテンツは取得されない場合がある）
- 記事本文が主要コンテンツとなるURLを選ぶ
- ログイン不要のパブリックURLを使用する
- 複数URLを使う場合は内容の重複を確認

**注意点:**
- JavaScript依存のSPA（Single Page Application）は内容取得が不完全な場合がある
- ペイウォール付きコンテンツはアクセスできない
- サイトの構造によってはナビゲーション等のノイズが含まれる

**品質チェック:**
```
□ URLがパブリックにアクセス可能か
□ ページの主要コンテンツが静的HTMLか
□ ログイン不要か
□ コンテンツが本文中心か（広告・サイドバー比率が低いか）
```

---

### YouTube

**強み:** 音声・映像情報をソースとして活用可能

**最適化ポイント:**
- 字幕（CC）付きの動画を優先する — 自動生成字幕より手動字幕の方が精度が高い
- 公開後72時間以上経過した動画が安定して処理される
- 短い動画（5-30分）が最も効果的
- プレイリストではなく個別の動画URLを指定

**注意点:**
- 非常に長い動画（2時間超）は処理に時間がかかる
- 音質が悪い動画はトランスクリプト精度が低下
- 音楽中心のコンテンツは有用な情報が少ない
- 年齢制限付き・非公開動画はアクセス不可

**品質チェック:**
```
□ 字幕（CC）が利用可能か
□ 音声品質が十分か（雑音が少ないか）
□ スピーカーの発話が明瞭か
□ 公開動画か
```

---

### Audio Files

**強み:** インタビュー、ポッドキャスト、会議録音をソース化

**最適化ポイント:**
- トランスクリプト付きのファイルがあれば併用する
- 高品質な録音（モノラル/ステレオ、ノイズ低減済み）を使用
- 複数話者の場合、話者識別が容易な録音が望ましい

**注意点:**
- 対応フォーマット: MP3, WAV等（最新の対応状況を確認）
- ファイルサイズ制限あり
- 音声のみのため、視覚的な情報（スライド等）は別途ソースとして追加が必要

**品質チェック:**
```
□ 音声が明瞭か
□ 背景ノイズが最小限か
□ ファイルサイズが制限内か
□ トランスクリプトが利用可能な場合は併用するか
```

---

### Copied Text / Pasted Content

**強み:** 最も柔軟。複数ソースから必要部分だけを集約可能

**最適化ポイント:**
- 見出しを付けて構造化する（プレーンテキストよりMarkdown風の構造が有効）
- 出典を明記する（NotebookLMが参照元を追跡できるように）
- 最重要情報を先頭に配置する
- 1つのノートは1つのトピックに集中させる

**注意点:**
- 文字数制限あり（ノートあたり）
- フォーマットは簡素化される
- 画像は貼り付け不可

**品質チェック:**
```
□ テキストに見出し構造があるか
□ 出典が明記されているか
□ 1ノート1テーマに整理されているか
□ 文字数制限内か
```

## Notebook Composition Patterns

### Pattern 1: Single Deep

**概要:** 1つの大きなソースを徹底的に掘り下げる

```
ノートブック構成:
├── [メインソース] (1つの包括的なドキュメント)
└── [補助ノート] (用語定義やコンテキスト情報、任意)
```

**最適なケース:**
- 1冊の書籍・論文を深く分析したい
- 長い報告書の要約を作りたい
- 単一トピックの深い理解を得たい

**ステアリングのコツ:**
- ソースが1つなので、フォーカスポイントを明確にする
- 「特にこの章/セクションに注力して」という指示が効果的
- 全体をカバーしようとせず、最も重要な部分に絞る

---

### Pattern 2: Multi-Perspective

**概要:** 同じトピックについて異なる視点のソースを3-5本集める

```
ノートブック構成:
├── [ソースA] (視点1: 例えば技術的な分析)
├── [ソースB] (視点2: 例えばビジネスインパクト)
├── [ソースC] (視点3: 例えばユーザー体験)
└── [ソースD] (視点4: 例えば批判的な見解、任意)
```

**最適なケース:**
- 多角的な分析が必要なトピック
- The Debate形式のAudio Overviewに最適
- 偏りのない包括的な理解を得たい

**ステアリングのコツ:**
- 各ソースの視点の違いを明示する
- 「各視点を公平に取り上げて」or「Xの視点を重視して」を指定
- ソース間の矛盾点を積極的に議論させる指示が有効

---

### Pattern 3: Hierarchical

**概要:** 概要→詳細の階層構造でソースを配置

```
ノートブック構成:
├── [概要ソース] (全体像を示す1本)
├── [詳細ソースA] (サブトピック1の深掘り)
├── [詳細ソースB] (サブトピック2の深掘り)
└── [詳細ソースC] (サブトピック3の深掘り、任意)
```

**最適なケース:**
- Lecture Mode（概要→詳細の教育フロー）
- Slide Deck（全体像→各セクション深掘り）
- 新しいトピックを体系的に学びたい

**ステアリングのコツ:**
- 「まず全体像から始めて、次に各トピックを深掘りして」が自然
- 概要ソースをベースに、詳細ソースで補強する構成
- 詳細に入る前に必ず概要に触れるよう指示

---

### Pattern 4: Comparative

**概要:** 2-3つの対象を比較するためのソース構成

```
ノートブック構成:
├── [対象Aのソース] (技術/製品/アプローチA)
├── [対象Bのソース] (技術/製品/アプローチB)
└── [比較フレームワーク] (比較軸を定義するノート、任意)
```

**最適なケース:**
- 技術選定・ツール比較
- Infographic（比較分析）の生成
- The Critique（製品評価）の素材

**ステアリングのコツ:**
- 比較軸（基準）を明示する指示が重要
- 「AとBを以下の観点で比較して: [基準リスト]」
- 勝者を決めるか、使い分けを提案するか、出力の方向性を指定

---

### Pattern 5: Chronological

**概要:** 時間軸に沿ったソースで変遷・進化を追う

```
ノートブック構成:
├── [初期のソース] (起源・初期段階)
├── [中期のソース] (発展・転換期)
├── [最新のソース] (現状・最新動向)
└── [将来展望] (予測・ロードマップ、任意)
```

**最適なケース:**
- 技術の進化・歴史を語るAudio Overview
- 業界トレンドの分析
- プロジェクトの振り返り・レトロスペクティブ

**ステアリングのコツ:**
- 「時系列で変遷を追って」という構造指示
- 各時期の「なぜ変わったか」の因果関係に注力させる
- 「過去→現在→未来」の自然な物語構造を活用

## NotebookLM Constraints Reference

### General Limits

| Constraint | Value | Notes |
|-----------|-------|-------|
| Max sources per notebook | 50 | 推奨は5-15本 |
| Max source size | Varies by type | PDFは大きなファイルに対応 |
| Max notebooks | Unlimited (Plus), Limited (Free) | |
| Supported source types | PDF, Google Docs, Google Slides, Web URL, YouTube, Audio, Copied Text | |

### Audio Overview Limits

| Constraint | Free Tier | Plus Tier |
|-----------|-----------|-----------|
| Generation per day | Limited | Higher limit |
| Customization | Basic | Full (including Background mode) |
| Interactive mode | Limited | Full |

### Video Overview Limits

| Constraint | Free Tier | Plus Tier |
|-----------|-----------|-----------|
| Availability | Limited or N/A | Available |
| Visual styles | Limited | All 8 styles |
| Length options | Limited | Full range |

### Other Format Limits

| Format | Free Tier | Plus Tier |
|--------|-----------|-----------|
| Slide Deck | Available | Enhanced |
| Infographic | Available | Enhanced |
| Mind Map | Available | Enhanced |
| Deep Research | Limited | Full |

**Note:** NotebookLMの機能と制限は頻繁に更新される。最新の公式情報を確認することを推奨。

## Free vs Plus Feature Comparison

| Feature | Free | Plus |
|---------|------|------|
| Notebooks | Limited | More / Unlimited |
| Sources per notebook | 50 | 50 |
| Audio Overview | Basic | Full + Background |
| Video Overview | Limited | Full (all styles) |
| Slide Deck | Available | Enhanced |
| Infographic | Available | Enhanced |
| Mind Map | Available | Enhanced |
| Deep Research | Limited | Full |
| Interactive Audio | Limited | Full |
| Priority processing | No | Yes |

**Prism's Recommendation:**
- Free tier でも基本的なAudio Overview と Slide Deck は十分に活用可能
- Video Overview の全スタイル活用にはPlusが有効
- 頻繁に使用する場合はPlus tier が時間効率で優位

## Universal Source Quality Checklist

すべてのソースタイプに共通する最終チェック:

```
## Pre-Upload Checklist

### Content Quality
□ 情報が正確で最新か
□ 出典が信頼できるか
□ ソース間で矛盾がないか（ある場合は意図的か）
□ 機密情報・個人情報が含まれていないか

### Structure
□ 見出しやセクション分けが明確か
□ 最重要情報がドキュメントの前半にあるか
□ 不要なボイラープレート（免責事項等）を除去したか
□ テキストが機械読み取り可能か

### Optimization
□ ソース数が目的に対して適切か（多すぎ/少なすぎないか）
□ 各ソースが明確な役割を持っているか
□ ノートブック構成パターンを選択したか
□ 言語が統一されているか
```
