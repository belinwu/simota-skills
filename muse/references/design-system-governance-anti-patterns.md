# Design System Governance & Scaling Anti-Patterns

> デザインシステムの採用失敗、ガバナンスドリフト、バリアントスプロール、スケーリングの落とし穴

## 1. 採用失敗 6 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|------|
| **DS-01** | **過剰エンジニアリング** | 実需なしに大量コンポーネントを先行構築 | 使用率の低いコンポーネント多数、開発リソース浪費 | 実際のニーズから逆算、3+ プロダクトで使用見込みのあるもののみ |
| **DS-02** | **オーナー不在** | 専任管理者なしで放置 | ドキュメント陳腐化、更新停止、貢献方法不明 | 専任チームまたは明確な RACI 定義 |
| **DS-03** | **トップダウン強制** | 協調なしに上から押し付け | 一部チームのみ採用、他チームは回避 | 各チームからのフィードバック収集、共同設計 |
| **DS-04** | **ドキュメント分散** | デザイン/コード/ドキュメントが別場所 | 正しい情報源が不明、重複実装 | 単一ソース（Storybook + コード = 真実） |
| **DS-05** | **複雑性ミスマッチ** | 過度に抽象化された設定オプション | 学習コスト高、チームが再構築を選択 | 80% のユースケースをシンプルに、20% を拡張で対応 |
| **DS-06** | **ビジュアルライブラリ誤認** | デザインシステムを UI 部品集と誤解 | 戦略的価値の欠如、横断的採用の失敗 | 組織インフラとして位置づけ、開発/デザイン/ビジネス横断 |

## 2. ガバナンスドリフト 5 パターン

```
ドリフトパターン:

  Drift-01: バリアントスプロール
    原因: 「今回だけ」の例外が蓄積
    兆候: 同一コンポーネントに 10+ バリアント
    対策: 新バリアント提案にユースケース証拠必須

  Drift-02: トークンオーバーライド
    原因: ローカルで spacing/color/typography を上書き
    兆候: !important 使用増加、トークンカバレッジ低下
    対策: オーバーライド検出の自動監査、月次レポート

  Drift-03: リビルドトランスレーション
    原因: エンジニアがデザインをゼロから再構築
    兆候: デザインと実装の乖離、未ドキュメントの置換
    対策: プロダクションコンポーネントでプロトタイプ構築

  Drift-04: 例外クリープ
    原因: 一時的例外に期限設定なし
    兆候: 例外が恒久化、移行パス未定義
    対策: 例外にサンセット日を設定、移行計画を添付

  Drift-05: シャドウコンポーネント
    原因: チームが独自にコンポーネントを再作成
    兆候: 類似コンポーネントの重複、不信感
    対策: 貢献フローの簡素化、ニーズの早期吸い上げ
```

## 3. ガバナンスモデル比較

| モデル | 特徴 | 強み | 弱み | 適用規模 |
|--------|------|------|------|---------|
| **集権型** | 単一チームが全決定 | 一貫性高、責任明確 | ボトルネック、入力限定 | 小〜中規模 |
| **連合型** | 複数チームがガイドライン下で貢献 | 柔軟、ボトルネック回避 | 一貫性リスク、調整コスト | 中〜大規模 |
| **コミュニティ型** | オープン貢献 + モデレーション | 包括的、採用促進 | 品質課題、意思決定遅延 | 大規模 |
| **ハイブリッド型（推奨）** | コアチーム + 分散貢献者 | バランス、柔軟 | コミュニケーション複雑 | 中〜大規模 |

## 4. スケーリング時の品質ゲート

```
10 ステップ意思決定フレームワーク:

  1. 既存ソリューションで解決可能か確認
  2. デザインシステムチームに代替案を相談
  3. 変更タイプの分類（一時的 vs システム追加）
  4. プロトタイプ + テストで検証
  5. DS チーム + プロダクトチームによるピアレビュー
  6. UX + 開発の実現可能性検証
  7. 横断チームによる最終レビュー
  8. ドキュメント化 + バージョニング + リリース計画
  9. 段階的ロールアウト + ステークホルダー通知
  10. 品質保証（visual regression + a11y テスト）

測定メトリクス:
  - リビルド/手戻り時間
  - システム使用率（採用率）
  - バリアントスプロール数/月
  - トークンオーバーライド頻度
  - 例外の経過日数
  - QA ドリフトインシデント数
```

## 5. 採用促進のベストプラクティス

```
文化 > コンポーネント:
  - 技術的完成度より文化的浸透が持続性を決定
  - 「使わなければならない」ではなく「使いたくなる」設計
  - 成功事例の共有、採用チームの表彰

漸進的採用戦略:
  Level 1: トークンのみ採用（色・余白の統一）
  Level 2: 基本コンポーネント採用（Button, Input, Card）
  Level 3: パターン採用（Layout, Form, Navigation）
  Level 4: 完全統合（全コンポーネント + ガバナンス参加）

貢献フローの簡素化:
  ❌ 30 ステップの提案プロセス
  ✅ Issue テンプレート → ピアレビュー → 実装 → リリース
  ❌ コアチームのみが変更可能
  ✅ 明確なガイドライン + セルフサービス貢献
```

## 6. Muse との連携

```
Muse での活用:
  1. SCAN フェーズで DS-01〜06 とドリフトパターンのスクリーニング
  2. トークンカバレッジ監査でオーバーライド検出
  3. バリアント数の定期モニタリング
  4. 新トークン/コンポーネント提案時のガバナンスチェック

品質ゲート:
  - バリアント 10+ → 統合提案（Drift-01 防止）
  - !important 使用 → トークンオーバーライドの根本原因調査（Drift-02 防止）
  - 類似コンポーネント発見 → 統合または差別化の明確化（Drift-05 防止）
  - ドキュメント更新日 > 1 スプリント → 更新リクエスト（DS-04 防止）
  - 使用率 < 10% のコンポーネント → 削除検討（DS-01 防止）
```
