# Design Review & Feedback

> デザインレビューの進め方、フィードバックフレームワーク、クリエイティブディレクションのアンチパターン

## 1. デザインレビュー 7 大アンチパターン

| # | アンチパターン | 問題 | 影響 | 対策 |
|---|-------------|------|------|------|
| **DR-01** | **目的なきレビュー** | ペルソナ/目標の合意なしでレビュー開始 | 主観的フィードバックの応酬 | レビュー前に問題定義・ユーザーニーズ・目標を共有 |
| **DR-02** | **曖昧フィードバック** | 「もっとポップに」「いい感じに」 | デザイナーが解釈不能 | 具体的要素の指摘 + 理由 + 目標との関連 |
| **DR-03** | **マイクロマネジメント** | ピクセル単位の指示を逐一 | デザイナーの自律性喪失、モチベーション低下 | 目標とフレームワークを示し、手段はデザイナーに委ねる |
| **DR-04** | **ネガティブのみ** | 問題点のみ指摘、良い点に言及なし | 防御的姿勢、心理的安全性の低下 | ポジティブ + ネガティブ + 改善提案の 3 点セット |
| **DR-05** | **即座の解決策提示** | 問題を聞く前に解決策を押し付け | 根本原因の見逃し、最適解の排除 | まず問題の理解 → 次にオプションの探索 |
| **DR-06** | **レビュー対象外の拡大** | スコープ外の議論に脱線 | 時間浪費、元の課題が未解決 | 事前にスコープ明示、ファシリテーターが軌道修正 |
| **DR-07** | **フィードバック無応答** | ステークホルダーが無反応 | デザイン進行停滞、方向性の確信なし | 期限付きレビュー依頼、デフォルト承認ルール |

## 2. フィードバックの 3 形態

```
3 Forms of Feedback（Stephen P. Anderson）:

  1. Reaction（反応）
     - 直感的・感覚的な反応
     - 「いいね」「好きじゃない」
     - 注意: レビュアー自身がターゲットユーザーでない場合、無効
     - 活用: アイスブレーカーとして短時間で収集

  2. Direction（方向性）
     - 「もっと〇〇に」「〇〇を減らして」
     - 注意: 理由なき方向性は設計目標との整合が不明
     - 活用: WHY を追加質問して意図を明確化

  3. Critique（批評）
     - 設計目標に対する達成度の分析
     - 「この配色はターゲットユーザーの信頼感構築に寄与するか？」
     - 最も有用: 目標ベースの客観的評価
     - 活用: デザインレビューの中核

推奨バランス:
  Reaction → 5%（冒頭のみ）
  Direction → 15%（方向性の確認）
  Critique → 80%（目標ベースの分析）
```

## 3. デザインレビュー進行ガイド

```
レビュー前:
  1. スコープ・アジェンダの配布
  2. デザインの事前共有（反応的フィードバック防止）
  3. レビュー目標の明確化
     - 何を評価するか（ビジュアル/UX/ブランド適合/技術実現性）
     - 何を評価しないか（スコープ外の明示）

レビュー中:
  1. プレゼンター: 背景・目標・ユーザーコンテキストの説明
  2. プレゼンター: デザインのストーリーを語る（決定の WHY）
  3. ファシリテーター: フィードバック収集
     - Round Robin: 順番に全員が発言
     - Quotas: 「良い点 2 つ、改善点 2 つ」
  4. ファシリテーター: 目標に紐づける質問でリフレーム
     - ❌ 「このレイアウトは嫌い」
     - ✅ 「このレイアウトはユーザーのタスク完了をどう支援するか？」
  5. 公開ドキュメントでリアルタイム記録

レビュー後:
  1. アクションアイテムの整理（優先度・担当・期限）
  2. 参加者への議事録配布
  3. 次回レビューのスケジュール設定

推奨設定:
  - 時間: 30 分/週（定期開催）
  - 人数: 3-7 名
  - ファシリテーター: ローテーション制（スキル育成）
```

## 4. クリエイティブディレクションの原則

```
効果的なクリエイティブディレクション:

  1. Why-First（なぜ先行）
     - ビジュアル提示前にビジネス目標・ユーザー課題を説明
     - デザイン決定の根拠を常に明示
     - 「なぜこの方向か」が説明できない決定は保留

  2. Options, Not Orders（指示ではなくオプション）
     - 最低 3 方向のオプション提示
     - 各オプションのトレードオフを明示
     - 推奨案を示すが、決定は合意プロセスで

  3. Constraints as Creativity（制約を創造性に）
     - ブランドガイドライン・技術制約・予算を創造性の枠に
     - 「制約なき自由」は方向性喪失の原因
     - 制約を明示してから創造を開始

  4. Evidence-Based（エビデンスベース）
     - リサーチ/ペルソナ/データに基づく判断
     - 「好み」と「効果」を明確に区別
     - A/B テスト提案で主観を排除

  5. Delegation with Context（文脈付き委譲）
     - 実行エージェントに WHY + WHAT を伝え、HOW は委ねる
     - 成功基準を測定可能な形で定義
     - マイクロマネジメントの回避（DR-03 防止）
```

## 5. Vision との連携

```
Vision での活用:
  1. 全モードでレビューフェーズに 3 Forms of Feedback 適用
  2. Direction Doc 提示時に Why-First 原則を遵守
  3. Muse/Palette/Flow への委譲時に Delegation with Context 実践
  4. Echo による検証時に Critique 形式のフィードバックを要求

品質ゲート:
  - Direction Doc にデザイン決定の WHY が含まれているか
  - 3 方向以上のオプションが提示されているか
  - 委譲指示に成功基準が測定可能な形で定義されているか
  - フィードバックが Critique 形式（目標ベース）か
  - レビューアジェンダにスコープが明示されているか（DR-06 防止）
```
