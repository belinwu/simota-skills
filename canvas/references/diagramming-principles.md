# Diagramming Design Principles

> C4 実践知見、UML ベストプラクティス、DFD 設計、ER 図、依存関係グラフ、ジャーニーマップ

## 1. C4 モデル実践知見

### Level 1-2 で十分な場合が多い

| レベル | 対象読者 | メンテナンスコスト |
|--------|---------|------------------|
| Level 1 (Context) | 全ステークホルダー | 低 |
| Level 2 (Container) | 技術チーム | 中 |
| Level 3 (Component) | 開発者 | 高 (コードと重複しやすい) |
| Level 4 (Code) | 必要時のみ | 非常に高 |

### 実践上のベストプラクティス

1. **全レベルを使う必要はない**: ほとんどのチームには Level 1 + Level 2 で十分
2. **要素に必ずメタデータ**: 名前・種類・技術選定・説明文
3. **凡例を必ず含める**: 図形、色、略語の意味を明示
4. **Level 2 図に番号付きリレーションシップ**: 別途 Dynamic Diagram を作るより保守コストが低い
5. **反復的に精製**: 理解が深まるにつれて段階的に改善

### 補助ダイアグラム

| 補助図 | 用途 |
|--------|------|
| System Landscape | 複数システムのコンテキスト俯瞰 |
| Dynamic Diagram | コラボレーション/シーケンスの動的表現 |
| Deployment Diagram | インフラストラクチャとランタイム環境 |

### 落とし穴

- 組織構造と一致しないコンテキスト定義
- 自明な要素のモデリング (非自明なアーキテクチャ決定にフォーカスすべき)
- メンテナンス不可能な過度の詳細

## 2. UML ベストプラクティス

### シーケンス図

- Combined Fragments (`alt/opt/loop`) で制御フローを表現
- 1図のメッセージは **15-20 以下** に抑える
- アクターとシステム境界を明確に区別
- 並行処理は並列の垂直破線で表現

### クラス図

- 関連・集約・合成の使い分けを厳密に
- 重要な関係性のみを図示 (全メソッドを含めない)
- パッケージ図と組み合わせて大規模システムを階層整理

### ステートマシン図

- 複雑な場合は**階層型ステートマシン** (サブステート) を使用
- 状態数が多すぎる図は分割
- 初期状態と終了状態を明示
- ガード条件を適切に使用

### UML 全般

- **1 図 = 1 関心事** (混在させない)
- 適切な図の種類を選択:
  - 状態ライフサイクル → State Diagram
  - オブジェクト間メッセージ → Sequence Diagram
  - プロセスフロー → Activity Diagram

## 3. データフロー図 (DFD)

### 設計ルール

1. **Context 図から開始**: Level 0 でスコープ定義 → 詳細レベルへ
2. **プロセス数**: 1 DFD に **3-9 プロセス** が推奨最大
3. **データフローに集中**: 制御フロー (条件分岐・シーケンス) ではなくデータの移動
4. **表記法統一**: Gane-Sarson または Yourdon のいずれか

### 構成要素

| 要素 | 説明 |
|------|------|
| 外部エンティティ | システム外のデータソース/シンク |
| プロセス | データの変換処理 |
| データストア | データの保存場所 |
| データフロー | データの移動経路 (矢印) |

### モダン DFD

- **論理 DFD と物理 DFD の分離**: 論理=ビジネスプロセス、物理=技術実装
- **ステークホルダーレビュー**: ビジネス、開発、エンドユーザーの3者で確認

## 4. ER 図のベストプラクティス

### 設計原則

1. **Crow's Foot 表記** (最も普及) を統一使用
2. **命名規則標準化**: snake_case、単数名詞テーブル名、略語禁止
3. **M:N には中間テーブル**: データ冗長性回避
4. **CASCADE 設定**: 外部キー制約で孤立レコード防止

### 2025 年のツール

| ツール | 特徴 |
|--------|------|
| SchemaSpy | OSS、DB メタデータから HTML 生成 |
| dbdiagram.io | DSL でスキーマ定義、SQL 貼付で即レンダリング |
| database.build | AI でスキーマ自動生成 |

### モダン DB 対応

- JSON/XML スキーマとの補完が必要 (ER 図は構造化データ特化)
- Agile 統合: イテレーティブに洗練

## 5. 依存関係グラフ可視化

### レイアウトアルゴリズム選択

| アルゴリズム | 推奨場面 |
|-------------|---------|
| 力指向 (Force-directed) | 50-500 ノードの一般的なグラフ |
| 階層型 (Hierarchical) | 有向グラフ、ワークフロー、依存チェーン |
| マトリクスビュー | 大規模依存関係の俯瞰 |

### 設計原則

1. **抽象化レベル統一**: ファイル単位ではなくパッケージ単位でグループ化
2. **動的・静的解析併用**: 静的で完全グラフ、動的でランタイムパス補完
3. **インタラクティブ操作**: ズーム、フィルタ、折りたたみは大規模で必須
4. **条件付き依存は別レイヤー**: クロスコンパイル環境での爆発的複雑度を管理

## 6. ジャーニーマップ vs ユーザーフロー

| 観点 | ジャーニーマップ | ユーザーフロー |
|------|----------------|---------------|
| レベル | マクロ (チャネル横断) | マイクロ (単一プロダクト内) |
| 内容 | 行動、感情、思考 | アクション + システム応答 |
| 時間スパン | 日〜月 | 分〜時間 |
| リサーチ | フィールドスタディ | ユーザビリティテスト |

### ジャーニーマップの構成要素

1. **ペルソナ**: 対象ユーザー定義
2. **フェーズ**: Awareness → Consideration → Decision → Service → Loyalty
3. **感情曲線**: 各フェーズの感情状態マッピング
4. **タッチポイント**: 製品/サービスとの接点
5. **ペインポイント & オポチュニティ**: 改善機会

### 可視化テクニック

- **ワイヤーフロー**: ローファイワイヤーフレーム + フロー矢印
- **タスク図**: 特定タスクのステップ分解
- **統合マップ**: ジャーニーのサブステージにユーザーフローを埋込

## 7. ADR (Architecture Decision Records) 可視化

### ベストプラクティス

- **1 ADR = 1 決定**: 複数決定の混在は分割のサイン
- **Markdown + リポジトリ格納**: コードと同じバージョン管理
- **インデックス作成**: README にリンク一覧
- **ドメインタグ**: `#cloud`, `#data`, `#integration` でフィルタリング

### 可視化手法

| 手法 | 用途 |
|------|------|
| フォースディレクテッドグラフ | ADR 間の接続を可視化 |
| タイムライン | 決定チェーンの時系列追跡 |
| C4 モデル統合 | Container Diagram から ADR へのリンク |

**テンプレート**: MADR (最普及)、Nygardian ADR、Y-Statements
