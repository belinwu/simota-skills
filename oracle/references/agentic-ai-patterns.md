# Agentic AI Patterns & Reliability

> エージェンティック AI のアーキテクチャパターン、信頼性ベストプラクティス、失敗モード、本番運用ガイドライン（2025-2026）

## 1. エージェント vs ワークフロー選択

```
判定フロー:

  タスクの性質は？
    │
    ├─ 固定シーケンス、予測可能な分岐
    │   → 決定的ワークフロー（Airflow / Temporal）
    │   利点: 強いリトライ保証、監査容易、デバッグ容易
    │
    ├─ データ条件で動的に分岐、ツール選択が必要
    │   → エージェント（LLM ベースの判断ループ）
    │   利点: 適応的計画、動的ツール選択
    │
    └─ 予測可能な段階 + 一部動的判断
        → ハイブリッド（固定ワークフロー + エージェント分岐）
        推奨: 予測可能な段階は固定、複雑性が要求する箇所のみエージェント

マルチエージェントへの関心: 2024 Q1→2025 Q2 で 1,445% 急増
ボトルネック: アルゴリズム的ではなくアーキテクチャ的
```

## 2. エージェントアーキテクチャパターン

| # | パターン | 構造 | 最適な場面 | リスク |
|---|---------|------|----------|-------|
| **AP-01** | **ReAct** | 思考 → 行動 → 観察 の反復ループ | 一般的な推論 + ツール使用 | 無限ループ、観察なしの推論ドリフト |
| **AP-02** | **Plan-and-Execute** | 計画フェーズ → 実行フェーズ分離 | 長いタスク、複数ステップ | 計画の硬直性、再計画コスト |
| **AP-03** | **Specialized Multi-Agent** | 責務別エージェント + オーケストレーション | 複合ドメイン、大規模タスク | ハンドオフ失敗、状態不整合 |
| **AP-04** | **Router** | 入力分類 → 専門エージェントへルーティング | 多様な入力タイプ | ルーティング誤分類 |
| **AP-05** | **Supervisor** | 中央エージェントが子エージェントを管理 | チーム型タスク | ボトルネック化 |

### Plan-and-Execute の優位性

```
単一ループ Reactive Agent vs Plan-and-Execute:

  Reactive: 各ステップで逐次判断 → 長いタスクで迷走しやすい
  Plan-and-Execute: 最初にタスクリスト生成 → 逐次実行
    → 反復的な行き来を削減
    → 長いタスクでのパフォーマンス向上（LangChain 分析）
    → 計画の明示化で監査・デバッグ容易

推奨: Plan-and-Execute をデフォルトとし、
      動的性が高いサブタスクのみ ReAct ループ
```

## 3. エージェント信頼性の 5 大原則

| # | 原則 | 説明 | 実装 |
|---|------|------|------|
| **1** | **構造化出力の強制** | 全出力に JSON スキーマを適用 | Structured Outputs 機能、スキーマ不適合時の自動修復 |
| **2** | **検証組み込み設計** | 最後ではなく各サブタスクで検証 | ステップごとの pass/fail チェック、Ground Truth 比較 |
| **3** | **不変の監査証跡** | 全ツールコールと判断根拠をログ | トレーシング SDK、immutable ログ |
| **4** | **最小権限アクセス** | エージェントをソフトウェアユーザーとして扱う | NIST AC-6 準拠、ロールベースツールアクセス |
| **5** | **コスト・レイテンシ上限** | 実行ごとのコスト・時間の上限設定 | Circuit Breaker、レート制限、バッチ処理 |

## 4. エージェント失敗モードと対策

| 失敗モード | 根本原因 | 検出方法 | 対策 |
|-----------|---------|---------|------|
| **Format Drift** | 確率的出力が構造を逸脱 | スキーマバリデーション | Structured Outputs 強制 |
| **Plan Divergence** | 無制限推論で目的から逸脱 | ステップ数監視、目標確認 | ReAct パラダイム（推論 + 観察） |
| **Ambiguity Loops** | 曖昧なパラメータで判断を繰り返す | ループ回数検出 | 明示的スキーマでツール選択を誘導 |
| **Silent Errors** | 検証レイヤー不在で失敗が伝播 | 出力品質のサンプリング検査 | 検証組み込み計画 |
| **Tool Abuse** | 不適切なツール選択や誤用 | ツールコール監査ログ | ツール使用ポリシー + 権限制限 |
| **Cost Explosion** | 再帰的ツールコールや無限ループ | コスト監視、ステップ上限 | 実行ごとのコスト上限 + Circuit Breaker |
| **State Corruption** | マルチエージェント間の状態不整合 | 状態スナップショット比較 | 明示的状態管理 + トランザクション |

## 5. マルチエージェント設計原則

```
責務分離の効果:
  - 信頼性向上: 障害を分離し、個別に対処可能
  - デバッグ容易性: 問題のあるエージェントを特定可能
  - スケーラビリティ: エージェント単位での調整・置換

4 つの設計原則:

  1. 単一責務: 各エージェントは 1 つの明確な責務
  2. 明示的インターフェース: エージェント間通信は構造化データ
  3. 障害分離: 1 エージェントの失敗がシステム全体に波及しない
  4. オーケストレーション層: 中央管理による協調（分散判断より安全）

アンチパターン:
  - 全能エージェント: 1 つのエージェントに全責務
  - 暗黙の通信: 自然言語のみでエージェント間通信
  - 障害伝播: サブエージェント失敗でシステム全体停止
  - 分散判断: 中央管理なしで各エージェントが独自判断
```

## 6. 本番運用チェックリスト

### 本番投入前

- [ ] 目的が明確で測定可能
- [ ] ツール一覧と安全なデフォルト値・パラメータ制約を定義
- [ ] 全ステップがバリデーション可能な成果物を出力（JSON スキーマ定義済み）
- [ ] 単一ソースの監査証跡（ログ/トレース）
- [ ] 障害時の Graceful Degradation（外部ステークホルダーへの影響最小化）
- [ ] スキーマ適合率監視を有効化
- [ ] NIST AC-6 準拠のアクセス制御
- [ ] コスト上限とレート制限を設定

### 継続的運用

- [ ] 定期的なモデル更新検証
- [ ] スキーマドリフト監視
- [ ] コスト傾向分析
- [ ] セキュリティ脆弱性スキャン
- [ ] バイアス・公平性監査
- [ ] 人間フィードバックの eval データセットへの還元

## 7. Prompt Caching の効果

```
エージェントでの Prompt Caching:

  効果:
    - レイテンシ最大 80% 削減
    - 入力トークンコスト最大 90% 削減

  適用箇所:
    - システムプロンプト（エージェント指示）
    - ツール定義（JSON スキーマ）
    - 共通コンテキスト（RAG ベースドキュメント）

  注意:
    - キャッシュヒット率はプロンプト安定性に依存
    - 動的コンテンツが多い場合、効果は限定的
    - キャッシュ無効化ポリシーの設計が必要
```

## 8. Oracle との連携

```
Oracle での活用:
  1. DESIGN モードで AP-01〜05 のパターン選択ガイド適用
  2. ASSESS モードで信頼性 5 大原則の適合性評価
  3. SPECIFY モードで本番チェックリストを Builder 仕様に含める
  4. EVALUATE モードでエージェント失敗モードの検出テスト設計

品質ゲート:
  - 構造化出力未定義 → DESIGN 段階でブロック（Format Drift 防止）
  - 各ステップのバリデーション未設計 → 検証組み込みを要求（Silent Errors 防止）
  - コスト上限未設定 → 実行前にコスト見積もり + 上限設定を要求（Cost Explosion 防止）
  - マルチエージェントで暗黙の通信 → 構造化インターフェース設計を要求
```
