# Realm Directory — HTML Prototype

Vanilla HTML/CSS/JS prototype for the agent ecosystem directory. Four pages, no
build step, no framework. Reads its data from `assets/data/agents.index.json`
(generated by Schema) and styles from `assets/tokens.css` (generated by Muse).

## Pages

| File | Layer | Purpose |
|---|---|---|
| `index.html`    | L1 | Hero + filter rail + grid grouped by category + legend footer |
| `category.html` | L2 | Category landing — roster, collaboration graph, "when to call" |
| `detail.html`   | L3 | Single-agent profile — overview / capabilities / collaboration / examples |
| `search.html`   | L1 | Sticky search bar + active filter tokens + relevance-sorted grid |

## Quick start

### Recommended (works everywhere)
```bash
cd ~/.claude/skills/realm/templates/realm-directory
python3 -m http.server 8000
# Visit http://localhost:8000/index.html
```

### `file://` (no server)
Open `index.html` directly in Chrome/Safari/Firefox. The app uses a synchronous
XHR fallback to load `assets/data/agents.index.json` into an inline
`<script type="application/json">` block when running over `file://`. If that
also fails (some browsers block `file://` XHR by default), the page shows a
graceful empty state and a hint to start a local server.

> Tip: Firefox is the most permissive with `file://`. Safari requires
> `Develop → Disable Local File Restrictions`. Chrome blocks it; use the local
> server.

### Keyboard shortcuts
- `/` — focus search input
- `Arrow keys / Home / End` — navigate the agent grid
- `Tab` — standard focus traversal; focus ring honors token `--realm-color-border-focus`

## Architecture decisions

### 1. Token-driven styling
`styles.css` references `tokens.css` custom properties only — no hard-coded
colors, spacings, or radii. Theme switching is just `[data-theme="dark"]` on
`<html>`; everything cascades. This means you can drop in a new tokens.css
revision and the prototype re-skins for free.

### 2. Index-driven rendering
`agents.index.json` is the single read source. Every page calls
`DataStore.load()` once, caches in memory, and renders. Detail pages also try
`agents/{name}.json` for richer fixture data, falling back to the index entry
on any error.

### 3. Filter state in the URL
`FilterState.fromUrl()` parses `?q=`, `?cat=`, `?cls=`, `?aff=`, `?sort=` on
boot; every chip toggle and search keystroke calls `toUrl()` (replaceState,
not pushState) so back/forward stays predictable and links are shareable.

Rule:
- **OR within an axis** (`?cat=ux-design,review` matches either)
- **AND across axes** (`?cat=ux-design&aff=saas` requires both)

### 4. Search scoring
Pure `String.includes()` against `name / archetype / category / class / tagline
/ description / capabilities[].summary` with a small weight ladder
(name === q → 100, prefix → 60, contains → 30, archetype → 12, …). No external
search lib. At 30 agents this scores in well under a millisecond; at 135 it
will still be sub-frame.

### 5. Portrait placeholder
Every card and the detail hero use `portraitSvg(agent)` which:
1. Reads `agent.portrait.status`. If `"pending"`, draws a small "PENDING" badge
   in the top-right corner.
2. Picks a class symbol from the `ClassSymbol` map (geometric primitives:
   star/hammer/ruler/eye/compass/etc) by `agent.class`.
3. Composites a 1:1 SVG with the symbol centered, the agent name in lowercase
   rounded sans below, and `archetype · category` as a sub-caption — exactly
   like the eventual portrait will read, so the UI never has to redraw the
   name when the real PNG arrives.

### 6. `file://` fallback
Each page contains an empty `<script type="application/json"
id="agents-index-fallback">` and a tiny inline IIFE that, on `file://`, runs a
synchronous XHR to fill it. `app.js` then prefers `fetch()` and falls back to
parsing the inline element. If both fail (Chrome `file://`), the user sees a
clear empty state with the local-server instruction.

## Replacing portraits

Sketch produces PNGs (720×720+, 1:1) and writes them to
`assets/portraits/{slug}.png`. To switch a card from placeholder to real:

1. Drop `assets/portraits/{slug}.png` into the directory (or symlink the
   shared `realm/portraits/` folder).
2. Edit `assets/data/agents/{slug}.json` — change `portrait.status` from
   `"pending"` to `"generated"`.
3. (Optional) Update `assets/data/agents.index.json` to mirror the change so
   the index-driven cards see it without a Schema rebuild.
4. Add an `<img>` rendering branch in `portraitSvg()`. Currently the function
   always renders the SVG placeholder; the easiest swap is:

```js
if (agent.portrait?.status === 'generated') {
  return `<img src="assets/portraits/${agent.name}.png"
              alt="${agent.name} portrait" loading="lazy" decoding="async" />`;
}
// else fall through to the existing SVG placeholder
```

Keeping the placeholder caption (`name · archetype · category`) baked into the
SVG matches what the real portrait will carry, so swapping does not regress
the layout.

## Scaling 30 → 135 agents

The current pipeline is linear in `agents.length`:

| Concern | At 30 | At 135 |
|---|---|---|
| Initial render | < 16 ms | ~50 ms |
| Search keystroke | < 1 ms | < 5 ms |
| `agents.index.json` size | ~50 KB | ~225 KB |

When the index passes ~200 KB or scoring lags on a keystroke:

1. **Split the index.** Generate `agents.index.lite.json` (name, class,
   category, tagline, description, capabilities summaries only) for
   search/list pages, and keep the full index for detail.
2. **Virtualize the grid.** Swap the section-by-category render for
   `IntersectionObserver`-based lazy mounting (only render visible cards).
3. **Add a service worker.** Cache `agents.index.json` and the image directory
   so revisits load instantly.
4. **Move scoring to a Web Worker.** The current `score()` function is pure;
   wrap the call site to post the query and receive a sorted array.

None of these require touching the page HTML — they live entirely in `app.js`
and the data layer.

## Known limitations

- Inline JSON fallback uses a synchronous XHR for `file://` boot; this is
  intentional and only runs when the user opened the file directly.
- Detail page's "See in map" link points to `../realm-map.html#{name}` — that
  template must add `id="{name}"` anchors to make it work; otherwise the link
  scrolls to the top.
- Only `agents.index.json` and `agents/*.json` are read at runtime. Schema
  files in `assets/data/schema/*.schema.json` are reference-only.
- Class symbols are geometric primitives, not the eventual Sketch portraits.
