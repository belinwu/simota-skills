# Rotation Patterns — 3軸×変換タイプ詳細ガイド

3軸（視野/視座/視点）それぞれの変換タイプ、適用条件、具体例、ドメイン別注釈。

---

## 軸1: 視野（Field / Scope）— どのスケールで見るか？

### 1-A: Zoom Out（スケールアップ）

**問い:** この問題を1つ上のシステムから見ると何が見えるか？

**適用条件:**
- 問題が局所最適化に陥っている兆候がある
- 「他部署/他システムには関係ない」という前提がある
- 短期視点で判断しようとしている

**変換例:**
```
元の問題: 「このAPIのレスポンスが遅い」
Zoom Out → 「ユーザーがその画面に到達するまでの全フローを見ると？」
→ さらに上 → 「このプロダクトが解決しようとしている根本的なユーザー体験は何か？」
```

**ドメイン別:**
- 技術: コンポーネント→サービス→アーキテクチャ→組織
- ビジネス: 施策→プロジェクト→戦略→市場全体
- 個人: タスク→プロジェクト→キャリア→人生設計

---

### 1-B: Zoom In（スケールダウン）

**問い:** この問題の最も小さな構成要素に分解すると何が見えるか？

**適用条件:**
- 問題が抽象的・概念的すぎて実行不能
- 「なんとなく上手くいっていない」という漠然とした不満がある
- 大きすぎる変更を一度に考えている

**変換例:**
```
元の問題: 「チームのコミュニケーションが悪い」
Zoom In → 「どの具体的な会議/やりとりで問題が起きているか？」
→ さらに下 → 「特定のあの発言がなぜ誤解を生んだか？」
```

---

### 1-C: Cross-Domain（ドメイン横断）

**問い:** 全く異なる分野で同じ構造を持つ問題はどう解決されているか？

**適用条件:**
- 同じアプローチを試し続けて詰まっている
- 業界の「常識」に縛られている兆候がある
- 創造的な打開策が必要

**変換例:**
```
元の問題: 「新機能のオンボーディング離脱率が高い」
Cross-Domain → 「ゲームのチュートリアル設計では離脱をどう防いでいるか？」
Cross-Domain → 「外科手術のトレーニングプログラムはどう段階を設計しているか？」
```

**注意:** 類似性はあくまでメタファー。実装の直接移植ではなく、構造の洞察として使う。

---

### 1-D: Temporal Zoom（時間軸変換）

**問い:** 5年後から振り返ると、今の判断はどう見えるか？ 5年前にはこれが存在しなかったとしたら？

**適用条件:**
- 緊急性バイアスが働いている（今が特別に感じる）
- 技術的負債vs速度のトレードオフ判断
- 「これは一時的な問題」という前提がある

**変換例:**
```
元の問題: 「今すぐモノリスをマイクロサービスに移行すべきか？」
Temporal Zoom（過去）→「3年前にこの議論をしていたら、今どうなっていたか？」
Temporal Zoom（未来）→「3年後、どちらの技術的負債を抱えていたいか？」
```

## 軸2: 視座（Standpoint / Vantage）— 誰の立場から見るか？

### 2-A: Role Shift（役割交換）

**問い:** この問題を最も影響を受けるステークホルダーの立場から見ると？

**適用条件:**
- 特定の役割（エンジニア/PM/ユーザー）の視点が支配的
- 「ユーザーは○○を求めている」という未検証の前提がある
- チーム間の対立がある

**変換例:**
```
元の立場: エンジニアとして「この機能の実装コストが高すぎる」
Role Shift → ユーザー:「この機能がないと何ができないか？」
Role Shift → ビジネス:「この機能がないと何の指標に影響するか？」
Role Shift → サポート:「この問題で何件問い合わせが来ているか？」
```

---

### 2-B: Hierarchy Shift（階層シフト）

**問い:** 意思決定者/現場担当者/顧客それぞれにとってこの問題はどう見えるか？

**適用条件:**
- 「上が理解していない」「現場がわかっていない」という摩擦がある
- 同じ言葉が違う意味で使われている（用語の齟齬）
- 優先度の判断が階層によって異なる

---

### 2-C: Timeline Shift（時間軸シフト）

**問い:** 過去の自分/未来の自分/次世代のユーザーにとって、この問題はどう見えるか？

**適用条件:**
- 「今だけ」の視点で設計している
- レガシーシステムとの向き合い方
- 意思決定の長期的影響が見えにくい

**変換例:**
```
元の視座: 現在の開発者として「このAPIは後方互換を壊してもいい」
Timeline Shift → 2年後の新入社員:「なぜここにこんな複雑な設定が必要なのか？」
Timeline Shift → 最初に設計した人:「当時の制約は何だったのか？」
```

---

### 2-D: Adversarial Shift（対立者視点）

**問い:** この計画に最も強く反対する人が見ている問題は何か？

**適用条件:**
- 計画が承認されやすい人だけで作られた
- 「なぜ反対するのか理解できない」という状況
- リスクの見落とし検証が必要

**変換例:**
```
元の計画: 「機能Aをデフォルトでオンにする」
Adversarial → プライバシー重視ユーザー:「同意なしに有効化されるのは問題」
Adversarial → サポートチーム:「設定変更の問い合わせが激増する」
Adversarial → 競合:「この機能はコピーしやすいからすぐ無効化される」
```

## 軸3: 視点（Lens）— どのフレームで見るか？

### 3-A: Systems Lens（システム思考）

**問い:** フィードバックループ・遅延・レバレッジポイントはどこにあるか？

**適用条件:**
- 「修正したら別の場所が壊れた」が繰り返される
- 施策の効果が想定外に拡大/縮小する
- 根本原因vs症状の区別が必要

**変換例:**
```
元の見方: 「バグの数が増えている（線形的な問題）」
Systems Lens → 「テストを後回しにする → 技術的負債増 → 開発速度低下 → 更なる後回し（強化ループ）」
→ レバレッジポイント: テスト文化の形成（ループを断つ）
```

---

### 3-B: Inversion Lens（反転思考）

**問い:** 「問題を解決する」ではなく「問題を最大化するにはどうすればいいか？」

**適用条件:**
- 従来の解決アプローチで詰まっている
- 「何をやるべきか」より「何をやめるべきか」を考えたい
- 前提に気づかず問題をループしている

**変換例:**
```
元の問い: 「ユーザーリテンションを上げるには？」
Inversion → 「ユーザーがすぐに離脱するにはどうすればいいか？」
→ 「分かりにくいUI、遅いロード、無関係な通知、価値のないコンテンツ」
→ 反転: これらを排除することがリテンション向上の本質
```

---

### 3-C: First Principles（第一原理）

**問い:** これを「当然のこと」と仮定せず、なぜそうあるべきかを最初から考えると？

**適用条件:**
- 「ずっとそうしてきたから」という理由で維持されているものがある
- 業界標準が真に最適かを疑いたい
- イノベーションの可能性を探索したい

**変換例:**
```
元の前提: 「コードレビューはPRマージ前に行う」
First Principles → 「レビューの目的は何か？（品質確保、知識共有、承認）」
→ 「全ての目的は『マージ前』でないと達成できないか？」
→ 「ペアプログラミング/自動チェックで一部代替できないか？」
```

---

### 3-D: Narrative Lens（物語フレーム）

**問い:** この問題を「誰かの旅」として語るとすれば、どんなストーリーか？

**適用条件:**
- データ・数値だけでは意思決定者の関心が引けない
- ユーザーの感情的体験が軽視されている
- 変化への抵抗を理解したい

**変換例:**
```
元の見方: 「オンボーディング完了率が62%（数値の問題）」
Narrative Lens → 「初めてアクセスしたユーザーが最初の10分間に感じること：
  希望（こんなことできそう）→ 混乱（何をすれば...）→ 挫折（やっぱり難しい）→ 離脱」
→ 感情の転換点を特定することで改善ポイントが変わる
```

---

## 変換の組み合わせパターン（推奨）

| 目的 | 推奨変換 |
|------|---------|
| 戦略立案 | Zoom Out + Timeline Shift + Systems Lens |
| イノベーション探索 | Cross-Domain + First Principles + Inversion Lens |
| 意思決定前の盲点確認 | Adversarial Shift + Inversion Lens + Timeline Shift |
| ユーザー体験改善 | Role Shift + Zoom In + Narrative Lens |
| 組織・チーム問題 | Hierarchy Shift + Systems Lens + Narrative Lens |
| 技術的判断 | Zoom Out + First Principles + Systems Lens |
