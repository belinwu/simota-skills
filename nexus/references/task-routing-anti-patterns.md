# Task Routing Anti-Patterns

> タスク分解の粒度ミス、ルーティング障害、役割混乱、動的委譲の罠、コンテキスト損失

## 1. タスク分解・ルーティング 7 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|---------|
| **TR-01** | **過剰分解** | 単純タスクを不要に細分化 | 計画オーバーヘッド、エージェント間のコンテキスト損失 | タスク複雑性の事前評価、単純タスクは分解スキップ |
| **TR-02** | **過少分解** | 複雑タスクを分解せず単一エージェントに丸投げ | 品質低下、ハルシネーション増加、タスク漏れ | 複雑性閾値を定義、超過時は自動分解 |
| **TR-03** | **役割混乱** | エージェントが指定外のタスクを実行 | 価格エージェントが契約承認、分析エージェントがセキュリティ検証 | 明示的な役割境界 + バリデーションチェックポイント |
| **TR-04** | **動的ルーティング暴走** | 全タスクを動的AI判断でルーティング | 予測不能な動作、ルーティングエラー蓄積 | 重要操作は静的マッピング、動的ルーティングは低リスクのみ |
| **TR-05** | **曖昧な委譲** | 「分析して対応してください」等の曖昧指示 | 非決定的な結果、エージェント間の動作不一致 | アクションスキーマで取り得る行動を限定的に定義 |
| **TR-06** | **階層間コンテキスト損失** | 多段分解でコンテキストが段階的に劣化 | Agent C が Agent A の意図の 60% しか受け取れない | 構造化コンテキスト伝搬 + 各段でのコンテキスト検証 |
| **TR-07** | **フォールバックルーティング濫用** | エラー時のフォールバックが不適切なエージェントへ転送 | 重要タスクを不適格なエージェントが処理 | 重要操作はフォールバックルーティング禁止、エスカレーション必須 |

## 2. タスク分解の粒度設計

```
粒度判断の決定木:

  タスク受信
    ├─ 複雑性評価（低/中/高）
    │   ├─ 低（単一ドメイン、明確な手順）
    │   │   → 分解不要、直接実行
    │   ├─ 中（2-3 ドメイン、定型手順）
    │   │   → 2-3 サブタスクに分解
    │   └─ 高（多ドメイン、非定型）
    │       → 階層的分解 + 計画フェーズ
    └─ 分解後の検証
        ├─ 各サブタスクが独立して検証可能か？
        ├─ サブタスク間の依存関係が明示的か？
        └─ 結果統合の方法が定義されているか？

  ❌ アンチパターン: 全タスクに同じ分解深度を適用
  ❌ アンチパターン: 分解後の統合方法を事前定義しない
  ✅ 推奨: 複雑性に応じた適応的分解
```

## 3. ルーティング精度の向上

```
仕様は API コントラクト:
  → 仕様の曖昧さが全障害の 41.77% の根本原因
  → 散文的な仕様ではなく、機械検証可能な制約を定義
  → 各エージェントの役割・能力・制約・成功基準をスキーマ化

役割境界の強制:
  1. 全プロンプトに明示的な役割境界と制限を記述
  2. 動的ルーティングの代わりに静的タスクマッピングを活用
  3. エージェントが適切なタスクのみ処理しているか検証
  4. 重要操作（承認・支払い等）のフォールバックルーティング禁止
  5. どのエージェントが何を処理したか監視 + 違反アラート

ルーティング信頼度とアクション:
  HIGH (≥0.85)   → 自動ルーティング
  MEDIUM (0.70-0.84) → ルーティング＋根拠表示
  LOW (0.50-0.69)  → 複数候補提示
  VERY_LOW (<0.50)  → エスカレーション（人間判断）
```

## 4. Discriminated Union パターン

```
アクションスキーマの設計（曖昧な委譲防止）:

  ❌ 悪い例:
    指示: "このイシューを分析して適切に対応してください"
    → エージェント A はイシューをクローズ
    → エージェント B はアサイン
    → エージェント C はエスカレーション
    → 非決定的で自動化に不適

  ✅ 良い例（Discriminated Union）:
    action:
      type: "close_issue" | "assign_to" | "escalate" | "add_label"
    → エージェントは定義済みアクションの中から「1つだけ」返す
    → 行動が予測可能、監査可能、テスト可能

  適用原則:
    1. 各ステップで取り得るアクションを列挙
    2. アクションごとに必須パラメータを定義
    3. 未定義アクションの実行をブロック
    4. アクション選択の根拠をログに記録
```

## 5. コンテキスト伝搬の設計

```
コンテキスト損失の防止:

  問題の数量化:
    → 3 エージェントチェーンで元の意図の 60% に劣化
    → トークン切り詰めが重要情報を無声で破棄
    → 8,000 トークン制限でコンテキストの 40% が失われるケースも

  構造化コンテキスト伝搬:
    1. 必須フィールドの定義（タスク目的、制約、成功基準）
    2. 各ハンドオフでのコンテキスト完全性チェック
    3. 要約ではなく構造化データでの伝搬
    4. コンテキストウィンドウ管理（選択的記憶）

  ❌ アンチパターン: 会話履歴全体を次のエージェントに渡す
    → トークンコスト爆発 + 注意力の分散

  ✅ 推奨: 構造化されたハンドオフテンプレート
    → 目的 + 完了済み作業 + 未解決事項 + 制約
```

## 6. Nexus との連携

```
Nexus での活用:
  1. CLASSIFY フェーズで TR-01/TR-02（粒度ミス）のスクリーニング
  2. CHAIN_SELECT で TR-03/TR-04（役割混乱・動的暴走）の防止
  3. EXECUTE フェーズで TR-06（コンテキスト損失）の監視
  4. agent-disambiguation.md と連携した TR-03 防止
  5. handoff-validation.md と連携した TR-06 防止

品質ゲート:
  - 単純タスク（低複雑性） → 分解スキップ（TR-01 防止）
  - 複雑タスク（高複雑性） → 計画フェーズ必須（TR-02 防止）
  - エージェント選択 → 役割境界の事前検証（TR-03 防止）
  - 重要操作 → 静的マッピング強制（TR-04 防止）
  - 委譲指示 → Discriminated Union 形式（TR-05 防止）
  - 多段チェーン → 各段でコンテキスト完全性チェック（TR-06 防止）
  - フォールバック → 重要操作はエスカレーション（TR-07 防止）
```
