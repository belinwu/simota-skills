# Requirements Writing Anti-Patterns

> PRD/SRS 作成の失敗パターン、仕様品質の罠、AI 時代の要件定義アンチパターン

## 1. 要件定義 7 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|------|
| **RW-01** | **長さ＝品質の錯覚** | 10K ワード超の PRD が何も伝えない | 読み手が「結局何を作るのか」不明、実装者が要点を見つけられない | PRD は 2-5 ページ、仮説駆動・メトリクス優先 |
| **RW-02** | **Non-Goals の欠如** | スコープ外を明示しない | スコープクリープ、関係者の期待不一致、実装範囲の膨張 | Non-Goals セクション必須、「やらないこと」を明記 |
| **RW-03** | **曖昧な成功基準** | 「ユーザビリティを改善」等の定量化不能な目標 | 完了判定不能、無限改善ループ、ステークホルダー間の認識齟齬 | 定量メトリクス必須（KPI + 目標値 + 計測方法） |
| **RW-04** | **実装の過剰規定** | PRD に技術実装詳細を書く | 開発チームの裁量を奪う、技術進化に追従不能 | PRD は「What/Why」に徹し、「How」は設計書に委譲 |
| **RW-05** | **戦略との断絶** | 機能要件だけで事業コンテキストなし | 優先順位判断不能、ビジネス価値不明、意思決定遅延 | ビジネスコンテキスト + 仮説 + 期待インパクト必須 |
| **RW-06** | **AI 生成肥大化** | AI に丸投げで大量の仕様テキスト生成 | 冗長・重複・一般論の羅列、プロジェクト固有性の欠如 | AI 出力は下書き、人間が編集・精査必須 |
| **RW-07** | **テスト不能要件** | 検証方法が定義されていない要件 | 受入テスト作成不能、「仕様通り」の判定不能 | 全要件に Given-When-Then 受入基準 |

## 2. 仕様品質の 6 つの基盤領域

```
高品質な仕様書の構成要素:

  1. コマンド・操作の明確化:
     → ユーザーが取り得る操作を網羅的に定義
     → 各操作の入力・出力・前提条件・事後条件
     → エラー時の振る舞い

  2. テスト戦略:
     → テスト種別（単体/結合/E2E）と対象範囲
     → テストデータの定義
     → カバレッジ目標

  3. プロジェクト構造:
     → ディレクトリ構成と命名規則
     → モジュール間の依存関係
     → 設定ファイルの役割

  4. コードスタイル:
     → 命名規則、フォーマット、リンター設定
     → エラーハンドリングパターン
     → ログ出力ガイドライン

  5. Git ワークフロー:
     → ブランチ戦略、コミットメッセージ規則
     → PR プロセス、レビュー基準
     → リリースフロー

  6. 境界定義（三層構造）:
     → Always: 常に従うルール
     → Ask first: 確認が必要な判断
     → Never: 禁止事項
```

## 3. AI 時代の仕様書アンチパターン

```
AI 支援ドキュメントの落とし穴:

  ❌ 指示の呪い（Curse of Instructions）:
    → 指示が多すぎるとパフォーマンスが低下
    → 300 行超のシステムプロンプトで精度劣化
    → 対策: 本当に重要なルールだけに絞る

  ❌ Vibe Coding（雰囲気コーディング）:
    → 「いい感じに作って」で仕様なし実装
    → 初期は速いが保守・拡張で破綻
    → 対策: Spec-Driven Development（仕様→計画→タスク→実装）

  ❌ コンテキスト過負荷:
    → 全ドキュメントを AI に丸ごと投入
    → トークン浪費 + 注意力分散
    → 対策: 構造化された必要最小限のコンテキスト

  ❌ 人間レビュースキップ:
    → AI 生成仕様をそのまま承認
    → プロジェクト固有の暗黙知が欠落
    → 対策: AI は下書き、人間が編集・承認

  ✅ 推奨: Spec-Driven Development
    → Specify（仕様定義）→ Plan（計画策定）→ Tasks（タスク分解）→ Implement（実装）
    → 各段階で人間のレビューゲート
```

## 4. モダン PRD の品質基準

```
良い PRD の 7 条件:

  1. 簡潔さ: 2-5 ページ（10K ワードは多すぎ）
  2. 仮説駆動: 「〇〇と仮定する。検証方法は△△」
  3. メトリクス優先: 定量的成功指標 + 計測方法
  4. Non-Goals 明記: やらないことを明示
  5. ビジネスコンテキスト: なぜ今これを作るか
  6. ユーザーストーリー: ペルソナ + シナリオ
  7. 受入基準: テスト可能な完了条件

  PMI データ:
    → プロジェクト失敗の 39% が不十分な要件/設計に起因
    → 要件定義の品質 = プロジェクト成功の最大の決定要因
```

## 5. Scribe との連携

```
Scribe での活用:
  1. UNDERSTAND フェーズで RW-01〜07 のスクリーニング
  2. prd-template.md と連携した RW-01/RW-04（肥大化・過剰規定）防止
  3. srs-template.md と連携した RW-03/RW-07（曖昧基準・テスト不能）防止
  4. writing-guidelines.md と連携した RW-02/RW-05（Non-Goals・戦略断絶）防止

品質ゲート:
  - PRD 5 ページ超 → 簡潔化を推奨（RW-01 防止）
  - Non-Goals セクション欠如 → 追加必須（RW-02 防止）
  - 定量メトリクスなし → KPI + 目標値を要求（RW-03 防止）
  - 技術実装詳細を含む PRD → 設計書への分離（RW-04 防止）
  - ビジネスコンテキストなし → 戦略連携を要求（RW-05 防止）
  - AI 生成テキストの無編集 → 人間レビュー必須（RW-06 防止）
  - Given-When-Then なし → 受入基準追加必須（RW-07 防止）
```
