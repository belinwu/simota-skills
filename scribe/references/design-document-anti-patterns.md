# Design Document Anti-Patterns

> HLD/LLD 設計書の失敗パターン、構造設計の罠、要件からの断絶、対象読者ミスマッチ

## 1. HLD（High-Level Design）7 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|------|
| **HD-01** | **要件スキップ** | 要件定義を飛ばしていきなり設計 | 設計の手戻り、要件との不整合、実装後の大幅修正 | 要件 → HLD → LLD の順序厳守 |
| **HD-02** | **過剰エンジニアリング** | 将来の仮想要件に対する過剰設計 | 複雑性増大、開発遅延、メンテナンスコスト爆発 | 現在の要件に集中、拡張ポイントのみ設計 |
| **HD-03** | **スケーラビリティ無視** | 初期は動くが負荷で破綻する設計 | 性能問題の後発、アーキテクチャレベルの再設計 | NFR（非機能要件）を設計初期に検討 |
| **HD-04** | **コンポーネント間関係の不明確化** | モジュール間の依存・通信が未定義 | 結合度の暗黙的増大、変更影響の予測不能 | 明示的なインターフェース定義 + 依存関係図 |
| **HD-05** | **セキュリティ後付け** | セキュリティを設計完了後に追加 | セキュリティホール、後付け対策のコスト増大 | 脅威モデリングを HLD フェーズで実施 |
| **HD-06** | **ドキュメント品質軽視** | 図なし・更新なし・曖昧な記述 | 設計意図の伝達失敗、実装者の誤解 | 図表必須 + 定期更新 + レビュープロセス |
| **HD-07** | **対象読者ミスマッチ** | 技術詳細を PM に、概要をシニア Dev に | 読者が理解不能、意思決定に使えない | 対象読者を明記 + 読者に合わせた詳細度 |

## 2. LLD（Low-Level Design）6 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|------|
| **LD-01** | **SOLID 原則無視** | クラス設計が SOLID に違反 | God Class、変更の波及、テスト困難 | SOLID チェックリストの設計時適用 |
| **LD-02** | **デザインパターン不在** | 既知のパターンを再発明 | 一貫性のないコード、レビューの難航 | 適切なパターンの選定 + 選定理由の記録 |
| **LD-03** | **コード明確性の欠如** | 命名・構造が不明瞭な設計 | 実装者が設計意図を読み取れない | 命名規則 + コード例 + シーケンス図 |
| **LD-04** | **テスト戦略の欠如** | テスト方法が設計に含まれない | テスタビリティの低い実装、後付けテスト | テスト戦略セクション必須（単体/結合/E2E） |
| **LD-05** | **過剰な詳細** | 実装コードレベルの記述 | 設計書がコードの複製、メンテナンス二重化 | 「設計判断」に集中、実装詳細はコードに委譲 |
| **LD-06** | **HLD との断絶** | LLD が HLD の設計方針と矛盾 | アーキテクチャ一貫性の崩壊、技術的負債 | HLD → LLD のトレーサビリティ確保 |

## 3. 設計書の構造設計

```
HLD の必須セクション:

  1. システム概要:
     → 目的・スコープ・対象読者
     → ビジネスコンテキスト・制約条件

  2. アーキテクチャ:
     → システム全体図（C4 Context/Container レベル）
     → コンポーネント間の通信方式
     → データフロー

  3. 技術選定:
     → 選定理由 + 代替案 + トレードオフ
     → 各技術の役割と責務

  4. 非機能要件:
     → パフォーマンス目標（レイテンシ、スループット）
     → スケーラビリティ戦略
     → セキュリティ方針
     → 可用性・災害復旧

  5. デプロイメント:
     → インフラ構成
     → 環境定義（dev/staging/prod）
     → CI/CD パイプライン

LLD の必須セクション:

  1. クラス設計:
     → クラス図 + 責務定義
     → SOLID 原則の適用箇所

  2. シーケンス設計:
     → 主要フローのシーケンス図
     → エラーフローの明示

  3. データ設計:
     → DB スキーマ + ER 図
     → データアクセスパターン

  4. テスト設計:
     → テスト戦略 + カバレッジ目標
     → テスタビリティの設計上の考慮

  5. HLD トレーサビリティ:
     → HLD の各コンポーネントとの対応表
     → 設計判断の根拠
```

## 4. 設計品質の定量指標

```
PMI データ:
  → プロジェクト失敗の 39% が不十分な要件/設計に起因
  → 設計レビューの欠如は後工程コスト 10-100 倍に

設計書品質チェック:
  □ 対象読者が明記されている
  □ 全コンポーネントに責務定義がある
  □ インターフェースが明示的に定義されている
  □ NFR が数値目標で記述されている
  □ 代替案とトレードオフが記録されている
  □ HLD → LLD のトレーサビリティがある
  □ 図表が最新の設計を反映している
  □ セキュリティの脅威モデルが含まれている

  ❌ アンチパターン: 設計書を一度書いて更新しない
  ❌ アンチパターン: テキストのみで図表なし
  ✅ 推奨: Living Document + 定期レビュー + 図表必須
```

## 5. Scribe との連携

```
Scribe での活用:
  1. STRUCTURE フェーズで HD-01〜07 / LD-01〜06 のスクリーニング
  2. design-template.md と連携した構造チェック
  3. REVIEW フェーズで設計品質チェックリスト適用
  4. writing-guidelines.md と連携した対象読者適切性検証

品質ゲート:
  - 要件ドキュメントなし → HLD 作成前に要件確認（HD-01 防止）
  - 将来要件への過剰対応 → 現要件に集中（HD-02 防止）
  - NFR セクション欠如 → 追加必須（HD-03 防止）
  - インターフェース未定義 → 明示化必須（HD-04 防止）
  - 対象読者未記載 → 追加必須（HD-07 防止）
  - SOLID チェック未実施 → LLD レビューで検証（LD-01 防止）
  - HLD 参照なし → トレーサビリティ確保必須（LD-06 防止）
```
