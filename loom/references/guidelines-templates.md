# Guidelines.md Templates

Figma Make 用 Guidelines の構成テンプレート集。公式推奨のマルチファイル構造に準拠。

---

## Architecture: Multi-File Structure

Figma Make は `Guidelines.md` をエントリーポイントとして読み込み、参照先サブファイルを必要に応じて選択する。**単一ファイルではなくディレクトリ構造で管理する。**

```
guidelines/
├── Guidelines.md              # Entry point: DS overview + reading order
├── overview-components.md     # Available components list
├── overview-icons.md          # Icon system and naming
├── design-tokens/
│   ├── colors.md              # Semantic color tokens
│   ├── typography.md          # Typography scale
│   └── spacing.md             # Spacing & radii tokens
└── components/
    ├── Button.md              # Per-component guidelines
    ├── Card.md
    └── DataTable.md
```

### Key Principle: Less Is More

> "More context is not always better. Too much can confuse the LLM. Add only the most important rules."
> — Figma Developer Docs

Write as if teaching a **new engineer**: sequential, clear, direct. Move edge cases to component sub-files.

---

## Entry Point: Guidelines.md

The top-level file introduces the design system and instructs the AI on reading order.

```markdown
# [Project Name] Design Guidelines
<!-- Generated by Loom | Source: [commit hash] | Date: YYYY-MM-DD -->
<!-- Version: X.Y.Z | Scope: [tokens|components|full] -->

## Design System Overview

This project uses [Design System Name].
Install: `npm install @scope/design-system` (if applicable)

### Available Components
See `overview-components.md` for the full list of available components.

### Available Icons
See `overview-icons.md` for icon naming and usage.

## Reading Order

When generating UI for this project:
1. Read this file first for overall context
2. Read `design-tokens/colors.md` and `design-tokens/spacing.md` for token values
3. Read relevant `components/*.md` files for specific component usage

## Global Rules

- Always use Auto Layout — never use manual positioning
- Always use semantic token names — never hardcode hex/px values
- Use Frames, not Groups
- Never use default layer names like "Frame 1" or "Rectangle 2"
- Use kebab-case for layer names: `header-nav`, `card-product`
- Use PascalCase for component names: `ProductCard`, `NavHeader`

## Explicit Prohibitions

- Do NOT use `className` prop on design system components
- Do NOT combine floating action buttons with bottom toolbars
- Do NOT nest Auto Layout deeper than 3 levels
- Do NOT use "Space between" mode — use explicit gap values
```

## Token Sub-Files

### design-tokens/colors.md

```markdown
# Color Tokens

## Philosophy
Colors use semantic naming: `{category}-{role}-{prominence}-{interaction}`.
For text/icons on colored backgrounds, use the `-on` variant.

## Decision Tree
- Background → `bg-*` tokens
- Text → `text-*` tokens
- Interactive elements → `accent-*` tokens
- Status feedback → `status-*` tokens

## Tokens

| Semantic Name | Light Value | Dark Value | Usage |
|--------------|-------------|------------|-------|
| `bg-primary` | `#FFFFFF` | `#1A1A2E` | Main background |
| `bg-secondary` | `#F5F5F5` | `#16213E` | Card/section background |
| `text-primary` | `#1A1A2E` | `#EAEAEA` | Body text |
| `text-secondary` | `#6B7280` | `#9CA3AF` | Helper text |
| `accent-primary` | `#3B82F6` | `#60A5FA` | CTA, links, focus rings |
| `accent-primary-hover` | `#2563EB` | `#93C5FD` | Hover state for primary accent |
| `border-default` | `#E5E7EB` | `#374151` | Dividers, card borders |
| `status-success` | `#10B981` | `#34D399` | Success states |
| `status-error` | `#EF4444` | `#F87171` | Error states |
| `status-warning` | `#F59E0B` | `#FBBF24` | Warning states |

## Pairing Rules
- On `accent-primary` background → use `text-on-primary` (#FFFFFF)
- On `status-*` background → use `text-on-status` (#FFFFFF) + icon

## Rules
- Always use semantic token names, never raw hex values
- Ensure 4.5:1 contrast ratio for text on backgrounds (WCAG AA)
- Status colors must be distinguishable without color alone (use icons)
```

### design-tokens/spacing.md

```markdown
# Spacing Tokens

## Scale (8px base grid)

| Token | Value | Usage |
|-------|-------|-------|
| `space-1` | 4px | Inline icon gaps (only exception to 8px grid) |
| `space-2` | 8px | Tight element spacing |
| `space-3` | 12px | Form field gaps |
| `space-4` | 16px | Standard content gaps |
| `space-6` | 24px | Section padding |
| `space-8` | 32px | Card padding |
| `space-10` | 40px | Section spacing |
| `space-12` | 48px | Major section dividers |
| `space-16` | 64px | Page section spacing |

## Border Radius

| Token | Value | Usage |
|-------|-------|-------|
| `radius-sm` | 4px | Buttons, inputs |
| `radius-md` | 8px | Cards, panels |
| `radius-lg` | 12px | Modals, dialogs |
| `radius-full` | 9999px | Pills, avatars |

## Shadows

| Token | Value | Usage |
|-------|-------|-------|
| `shadow-sm` | 0 1px 2px rgba(0,0,0,0.05) | Subtle lift (buttons) |
| `shadow-md` | 0 4px 6px rgba(0,0,0,0.1) | Cards, dropdowns |
| `shadow-lg` | 0 10px 15px rgba(0,0,0,0.1) | Modals, popovers |

## Rules
- Padding inside containers: minimum `space-4` (16px)
- Gap between sibling sections: minimum `space-6` (24px)
- Use `shadow-md` for elevated surfaces, `shadow-lg` for overlays only
```

```markdown
# Typography Tokens

## Scale

| Token | Size | Weight | Line Height | Usage |
|-------|------|--------|------------|-------|
| `heading-1` | 32px | 700 | 1.2 | Page titles |
| `heading-2` | 24px | 700 | 1.25 | Section titles |
| `heading-3` | 20px | 600 | 1.3 | Subsection titles |
| `heading-4` | 16px | 600 | 1.4 | Card titles |
| `body-lg` | 16px | 400 | 1.5 | Primary content |
| `body-md` | 14px | 400 | 1.5 | Secondary content |
| `body-sm` | 12px | 400 | 1.4 | Captions only |
| `label` | 14px | 500 | 1.2 | Form labels, tags |

## Rules
- Font family: [Project font stack]
- Maximum 2 font families per page
- Minimum body text size: 14px (never use body-sm for running text)
- Line height for body text: 1.5 minimum
- Heading hierarchy must be sequential (h1 → h2 → h3, no skipping)
```

## Component Sub-Files

### components/Button.md

```markdown
# Button

## When to Use
Use Button for user-initiated actions. Primary for the single main CTA per view.

## Structure
- Container: Auto Layout (horizontal, gap: `space-2`, padding: `space-2` `space-4`)
- Icon (optional): 20x20px, `Fill container` disabled
- Label: `label` token, center-aligned

## Variants

| Variant | Background | Text | Border |
|---------|-----------|------|--------|
| Primary | `accent-primary` | `#FFFFFF` | none |
| Secondary | transparent | `accent-primary` | `accent-primary` 1px |
| Ghost | transparent | `text-primary` | none |
| Danger | `status-error` | `#FFFFFF` | none |

## Sizes

| Size | Height | Font | Padding |
|------|--------|------|---------|
| Small | 32px | `body-sm` | `space-1` `space-3` |
| Medium | 40px | `body-md` | `space-2` `space-4` |
| Large | 48px | `body-lg` | `space-3` `space-6` |

## Auto Layout
- Direction: Horizontal
- Alignment: Center
- Resizing: Hug contents (width), Fixed (height per size)

## Do
- Use Primary for single main CTA per view
- Include icon + label for destructive actions

## Don't
- Don't stack more than 2 buttons inline
- Don't use Ghost variant for primary actions
- Don't use Small size for touch targets on mobile
```

## Version Management

All Guidelines files include version metadata:

```markdown
<!--
  Version: 1.2.0
  Generated: 2025-01-15T10:30:00Z
  Source commit: abc1234
  Scope: full
  Changes from 1.1.0:
    - Added status-warning token
    - Updated Button padding scale
    - Added DataTable component pattern
-->
```

| Change Type | Version Bump | Example |
|------------|-------------|---------|
| Token value change | PATCH (0.0.X) | `accent-primary` color adjusted |
| New component pattern | MINOR (0.X.0) | Added DataTable pattern |
| Structural reorganization | MAJOR (X.0.0) | Token naming scheme changed |

## Quality Checklist

Before delivery, verify:

- [ ] Entry-point `Guidelines.md` includes reading order instructions
- [ ] All tokens reference actual codebase values
- [ ] Component sub-files match existing code components
- [ ] Explicit prohibitions are specific and actionable (not vague)
- [ ] Total content is minimal — no redundant or rarely-needed rules
- [ ] Auto Layout rules specify direction, gap, padding, resizing
- [ ] Naming conventions align with codebase
- [ ] Version metadata is accurate
- [ ] No contradictions between files
