## Why This Matters in 2025-2026

The 2025 DORA Report establishes AI as an "amplifier" — it raises throughput and instability simultaneously. Empirical signals across 2024-2026 sources:

- AI-generated code contains security vulnerabilities at **29-45%** rates with **~20% hallucinated library references** (diffray 2025).
- Mutation kill rates on AI-generated code are **15-25% lower** than human-authored at equal line coverage (Meta FSE 2025).
- CodeRabbit 2025: AI-authored code shows **~1.7×** more issues than human-authored at review time.
- LLM-generated tests inflate raw coverage with **assertion-free or trivial-assertion** tests ("vibe testing", `dev.to/htekdev`).
- Developers spend **15-30 minutes per false positive** investigating AI-generated test failures (NUS 2024).

Vista does not block AI-generated tests — it surfaces the risk so teams can decide.

## AI-Origin Test Detection Signals

A test is flagged as **AI-Origin** when at least 2 of these signals match:

| Signal | Source | Threshold |
|--------|--------|-----------|
| Author matches LLM tool prefix | Git author or co-author trailer | `.*(claude|cursor|copilot|gpt|gemini|codeium|tabnine|aider).*` |
| Commit metadata contains LLM marker | `git log --grep` | `Co-Authored-By: Claude` / `[skip-claude]` / `Generated-By:` |
| Commit message LLM signature | `git log` body | `Generated by`, `via Claude Code`, `auto-generated` |
| Burst authorship | git stats | ≥10 tests added in a single commit by 1 author |
| File header AI marker | First 5 lines of test file | `// Generated by`, `// AI-assisted`, `# Generated:` |
| Test name pattern | Test ID lexical analysis | High n-gram similarity (>0.8) across many tests, mechanical naming patterns |

If only 1 signal matches → mark as **AI-Origin-Suspect** (lower confidence).

### When AI signals are absent

If no signals match, do not assume human-authored. Flag as **Origin-Unknown** in the dashboard with `n=<count>` so reviewers can decide whether to add author tracking discipline.

## Vibe Testing

**Definition:** AI-generated tests that exercise code without meaningfully verifying behavior — assertion-free, trivial getter/setter checks, or assertions that always pass.

### Detection (per test)

```yaml
VIBE_TESTING:
  signals_required: 2
  signals:
    - assertion_count: "< 1 per test"  # AST scan
    - assertions_are_trivial: "expect(x).toBeDefined() / toBeTruthy() with no value check"
    - mutation_kill_rate: "< 40%"
    - test_body_loc: "≤ 3 lines"
    - test_modifies_no_state: "no side-effect assertions"
```

### Output

```yaml
finding:
  id: AI-ORIGIN-VANITY
  test_id: "auth.session#getter_returns"
  signals: [ai_origin=2/6, assertion_count=0, mutation_kill_rate=18%]
  recommendation: "Manual review + Radar mutation strengthening; treat as untested for risk-weighted gap analysis"
```

## LLM Hallucination in Tests

**Definition:** Tests that reference non-existent APIs, libraries, fixtures, or functions — passing because they short-circuit before assertions.

### Detection

| Signal | Method |
|--------|--------|
| Imports a package not in `package.json` / `pyproject.toml` / `Cargo.toml` | Cross-reference imports vs manifest |
| Calls a function that does not exist in tested module | AST + symbol resolution |
| References a fixture/factory not defined anywhere | AST + glob over `__fixtures__` and `factories/` |
| Test passes only because import threw a caught error | Test execution + stack trace analysis |
| Mock has different shape than the real type | Type-check mock vs target type signature |

When ≥1 signal matches → emit `AI-HALLUCINATION` finding (this is high-severity, single-signal threshold is intentional).

## Snapshot Soup

**Definition:** Test suites where snapshots dominate over assertions, with low semantic value.

### Detection

```yaml
SNAPSHOT_SOUP:
  signals_required: 2
  signals:
    - snapshot_to_assertion_ratio: "> 0.6"  # snapshots / total assertions per file
    - unnamed_snapshot_ratio: "> 0.3"  # toMatchSnapshot() without identifier
    - snapshot_auto_update_frequency: "≥ 3 commits with --updateSnapshot in last 30 days"
    - snapshot_diff_volume: "median snapshot diff > 50 lines per change"
```

### Output

```yaml
finding:
  id: SNAPSHOT-SOUP
  scope: "tests/components/"
  signals: [ratio=0.71, unnamed=0.42, auto_update_commits=5]
  recommendation: "Replace snapshot tests with explicit assertions on user-visible properties; cite testing-library queries"
```

## Rework Rate (DORA 2025)

The 2025 DORA Report introduced **Rework Rate** as a new metric capturing AI-era instability — the share of changes that needed to be reverted, re-deployed, or hotfixed within a short window.

### Vista's contribution

Vista does not compute Rework Rate end-to-end (that's Pulse / Beacon territory). Vista contributes the **test-level rework signal**:

```yaml
test_rework_signal:
  - tests_modified_within_24h_of_creation: <count>
  - tests_deleted_within_7d_of_creation: <count>
  - tests_with_skip_marker_added_post_creation: <count>
  - tests_repeatedly_quarantined_and_unquarantined: <count>
```

Hand off these signals to Pulse via `VISTA_TO_PULSE_HANDOFF` for product-level Rework Rate computation.

## Detection Pipeline

For the `ai-lens` Recipe:

1. **INGEST**: git log + author metadata, test files, `coverage-final.json`, mutation results (if available), assertion AST scan.
2. **SCORE**: per-test AI-Origin signal count (0-6), assertion density, mutation kill rate.
3. **CLASSIFY**:
   - 0 signals → `Origin-Unknown`
   - 1 signal → `AI-Origin-Suspect`
   - ≥2 signals → `AI-Origin`
   - `AI-Origin` + (assertion_density < 1 OR mutation_kill_rate < 40%) → `AI-ORIGIN-VANITY`
   - Any test with hallucination signal → `AI-HALLUCINATION` (independent of origin)
4. **AGGREGATE**: per-file, per-author, per-day rollup.
5. **RENDER**: heatmap + table (see Visualization below).
