# Work Estimation Anti-Patterns

> LOC 生産性指標の問題、工数見積もりの落とし穴、逆インセンティブ、改善手法

## 1. LOC（コード行数）が生産性指標として失敗する 8 つの理由

| # | 理由 | 説明 | 影響 |
|---|------|------|------|
| **EA-01** | **解決策の複雑さ ≠ 問題の複雑さ** | 難しい問題ほど少ないコードで解ける場合がある | 熟練者が過小評価される |
| **EA-02** | **言語依存性** | 同じ機能: Java 50行 vs Python 10行 vs Haskell 3行 | 言語選択で「生産性」が変動 |
| **EA-03** | **リファクタリングペナルティ** | コード削減（=改善）がマイナスの生産性として記録 | リファクタリングの抑制 |
| **EA-04** | **非コーディング作業の無視** | 開発時間の 65-70% はコーディング以外（設計・調査・レビュー・デバッグ） | 重要な貢献が不可視化 |
| **EA-05** | **品質との無相関** | 行数が多い = バグが多い可能性 · テスタビリティと無関係 | 量を質と誤認 |
| **EA-06** | **コピペの奨励** | LOC最大化のためにコードの再利用を避ける逆インセンティブ | コード品質の劣化 |
| **EA-07** | **自動生成コードの混入** | ボイラープレート・設定ファイル・生成コードが集計に含まれる | 実際の労力と乖離 |
| **EA-08** | **チーム貢献の不可視化** | メンタリング・ペアプロ・設計レビューが LOC に反映されない | 協働文化の毀損 |

## 2. 工数見積もりの 6 大アンチパターン

| # | アンチパターン | 問題 | 対策 |
|---|-------------|------|------|
| **EA-09** | **楽観バイアス** | 「うまくいけば」の見積もり → 常に予定超過 | 過去データに基づく見積もり · バッファ係数（×1.5〜2.0） |
| **EA-10** | **アンカリング効果** | 最初の見積もりに引きずられる → 新情報を反映しない | 複数人独立見積もり → 平均値 · プランニングポーカー |
| **EA-11** | **パーキンソンの法則** | 見積もり時間いっぱい使う → 実際の複雑さと無関係に膨張 | バッファは明示的に分離 · 実績データとの比較 |
| **EA-12** | **スコープクリープ無視** | 初期見積もりにスコープ変更を反映しない → 「遅延」扱い | 見積もりのバージョニング · 変更時の再見積もり |
| **EA-13** | **個人の速度で標準化** | 最速の人の実績を基準にする → 大半のメンバーが「遅い」 | チーム平均 · パーセンタイル（P50/P75/P90）で範囲提示 |
| **EA-14** | **複雑さの線形仮定** | 2倍のコード = 2倍の時間 → 統合・テスト・デバッグの非線形コストを無視 | 複雑さ係数の段階的適用（既存ロジックの改善） |

## 3. 疲労と負の生産性

### 疲労による生産性逆転

```
研究に基づく知見:
  - 週 40 時間超の持続的稼働 → 1時間あたりの生産性低下
  - 週 60 時間超 → バグ導入率が急増 → 正味生産性がマイナス
  - 「負の生産性」: 疲労状態で書いたコードの修正コスト > 書かなかった場合のコスト
  - 深夜コミットのバグ率は日中の 1.5-2 倍（複数の研究で確認）

Harvest への示唆:
  - 稼働時間レポートで長時間労働を「高生産性」と表示しない
  - 深夜・週末コミットを警告として表示
  - 疲労リスクの注記を含める
```

## 4. Harvest 工数計算式の改善提案

### 現行式の問題点

```
現行: hours = (adds+dels)/100 × file_weight + files×0.25 + new_files×0.5

問題点:
  1. LOC ベース → EA-01〜08 の問題を内包
  2. レビュー・設計・デバッグ時間を含まない（EA-04）
  3. 自動生成コードを区別しない（EA-07）
  4. リファクタリング（大量削除）を過大評価する可能性
```

### 改善の方向性

```
改善案 1: 複雑さ補正
  complexity_factor を導入:
    - PR コメント数 × 0.1（レビューの往復回数で複雑さを推測）
    - PR 作成〜マージの日数 × 0.2（長い = 複雑）
    - ファイル種別の多様性 × 0.15（多種 = 統合コスト高）

改善案 2: 非コーディング時間の加算
  non_code_ratio = 0.4〜0.6（コーディングは全作業の 40-60%）
  total_hours = code_hours / (1 - non_code_ratio)
  → 注意: あくまで推定値であることを明記

改善案 3: レンジ提示
  min_hours = base_hours × 0.7
  expected_hours = base_hours
  max_hours = base_hours × 1.5
  → 単一値ではなくレンジで提示（EA-09 対策）
```

## 5. 見積もり精度の評価

### 精度メトリクス

```
Estimation Accuracy:
  Accuracy = 1 - |actual - estimated| / actual × 100%

  目標:
    Accuracy > 80% → 良好
    Accuracy 60-80% → 改善余地あり
    Accuracy < 60% → 見積もりプロセスの見直し

Bias Direction:
  Bias = (estimated - actual) / actual × 100%

  Bias > 0: 過大見積もり（バッファ過多）
  Bias < 0: 過小見積もり（楽観バイアス）
  目標: |Bias| < 20%
```

### 見積もり改善サイクル

```
Sprint N: 見積もり → 実行 → 実績記録
Sprint N+1: 実績データで精度評価 → バイアス方向を特定
Sprint N+2: 補正係数を適用 → 精度測定
Sprint N+3: 補正係数の安定化を確認

Harvest での自動化:
  - PR のサイクルタイム（実績）と工数見積もりの比較
  - チーム別の見積もりバイアスをトレンド表示
```

## 6. Harvest との連携

```
Harvest での活用:
  1. work-hours 計算結果に「推定値」であることを明記
  2. LOC を「生産性」として表示せず、「変更規模」として表示
  3. 工数レポートにレンジ（min/expected/max）を追加
  4. Individual レポートで稼働パターン警告を含める
  5. 見積もり精度のトレンドレポートを追加可能に

品質ゲート:
  - LOC を個人の生産性指標として使用しない（EA-01 防止）
  - 自動生成ファイルの除外ルールを適用（EA-07 防止）
  - 深夜・週末コミット比率 > 10% で疲労警告
  - 工数見積もり精度 < 60% でプロセス見直し提案
```
