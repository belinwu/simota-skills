# Reporting Anti-Patterns

> レポート生成のアンチパターン、メトリクスゲーミング、Goodhart の法則、改善手法

## 1. レポート設計の 8 大アンチパターン

| # | アンチパターン | 問題 | 対策 |
|---|-------------|------|------|
| **RA-01** | **データ過多** | 全メトリクスを羅列 → 意思決定に使えない | 対象読者に応じて 3-5 の核心メトリクスに絞る |
| **RA-02** | **コンテキスト欠如** | 数字だけ提示 → 「だから何？」 | 前期比較・目標との差異・トレンドを必ず併記 |
| **RA-03** | **チェリーピッキング** | 良い指標だけ強調 → 問題の隠蔽 | ポジティブとネガティブのバランスをルール化 |
| **RA-04** | **個人ランキング** | 個人のメトリクスを順位付け → チーム協働の破壊 | チーム単位の集計 · 個人データは本人のみ閲覧可能に |
| **RA-05** | **頻度過多** | 日次で全メトリクスを報告 → ノイズでトレンドが見えない | 指標ごとに適切な報告頻度を設定（日/週/月/四半期） |
| **RA-06** | **スナップショット偏重** | 単一時点のデータのみ → 改善/悪化の方向性が不明 | トレンドライン・移動平均を必ず含める |
| **RA-07** | **アクション不在** | 分析はあるが「次に何をすべきか」がない | 各指標に推奨アクションを紐付け |
| **RA-08** | **自動化の過信** | 機械生成を無検証で配信 → 異常値・誤分類が残る | 自動生成 + 人間レビューのハイブリッド運用 |

## 2. Goodhart の法則とメトリクスゲーミング

### Goodhart の法則

```
"測定が目標になると、良い測定ではなくなる"
  — Charles Goodhart (1975)

ソフトウェア開発での典型例:
  メトリクス: PR 数を測定
  → ゲーミング: 1つの機能を10個のPRに分割
  → 結果: PR 数は増えるが、統合コスト増加・レビュー負荷増大

  メトリクス: コードカバレッジ 80% を目標
  → ゲーミング: assert なしのテストでカバレッジだけ達成
  → 結果: カバレッジは上がるが品質は向上しない

  メトリクス: サイクルタイムの短縮
  → ゲーミング: レビューを省略して即マージ
  → 結果: 速度は上がるが変更失敗率が悪化
```

### ゲーミング検出パターン

| 兆候 | 考えられるゲーミング | 検出方法 |
|------|-------------------|---------|
| PR サイズが突然均一化 | 大きな変更を機械的に分割 | PR サイズの分散が縮小 + マージ間隔の異常な規則性 |
| レビュー時間が極端に短い | ラバースタンプ承認 | レビューコメント数 = 0 の承認率 |
| 金曜夕方のマージ急増 | 週次目標の帳尻合わせ | 曜日別マージ分布の偏り |
| テストカバレッジ急上昇 | アサーションなしテスト | テスト追加数 vs アサーション数の乖離 |
| コミット数の急増 | 些末な変更の量産 | コミットあたりの変更行数が急減 |

## 3. 報告頻度のガイドライン

### 指標別の適切な報告頻度

| 頻度 | 指標 | 理由 |
|------|------|------|
| **日次** | ビルド成功率 · デプロイ状況 · オープン PR 数 | 即時対応が必要な運用指標 |
| **週次** | サイクルタイム · マージ数 · レビュー応答時間 | フロー改善の PDCA サイクル |
| **月次** | DORA 4指標 · 品質トレンド · 工数分析 | トレンド分析に十分なサンプルサイズ |
| **四半期** | 技術的負債指標 · アーキテクチャ健全性 | 長期的な改善の方向性確認 |

### アンチパターン: 過少/過多の判断

```
報告が過多な兆候:
  - レポートを誰も読んでいない（開封率 < 30%）
  - 「前回と同じ」が連続 3 回以上
  - 数字の変動がノイズ範囲内（±5% 以内の振動）

報告が過少な兆候:
  - 問題が「気づかれないまま」悪化する
  - ステークホルダーが独自にデータを収集し始める
  - レトロスペクティブで「データがなくて議論できない」
```

## 4. 対象読者別レポート設計

### 3 層レポートアーキテクチャ

| 層 | 対象 | 内容 | 粒度 | 頻度 |
|----|------|------|------|------|
| **L1: Executive** | 経営層 · PM | ビジネスインパクト · デリバリー状況 | 高レベル KPI 3-5 個 | 月次/四半期 |
| **L2: Manager** | EM · TL | チームパフォーマンス · ボトルネック | チーム指標 + トレンド | 週次/月次 |
| **L3: Engineer** | 開発者 | 個人の貢献 · 技術的詳細 | PR レベルの詳細 | 日次/週次 |

### 各層で避けるべきこと

```
L1 (Executive) で避けるべき:
  ✗ 個別 PR の詳細
  ✗ 技術的な指標（コードカバレッジ、Lint エラー数）
  ✗ 個人名を含むランキング

L2 (Manager) で避けるべき:
  ✗ 個人の LOC ランキング
  ✗ コンテキストなしの数字
  ✗ 改善アクションのない問題提示

L3 (Engineer) で避けるべき:
  ✗ 他のメンバーとの比較
  ✗ ビジネス KPI のみ（技術的フィードバックなし）
  ✗ 過去データなしの単一スナップショット
```

## 5. 健全なレポートのチェックリスト

```
設計時チェック:
  □ 対象読者が明確か（L1/L2/L3）
  □ 核心メトリクスが 3-5 個に絞られているか（RA-01 防止）
  □ 各メトリクスにコンテキスト（前期比/目標比）があるか（RA-02 防止）
  □ ポジティブとネガティブの両方が含まれているか（RA-03 防止）
  □ 個人ランキングが含まれていないか（RA-04 防止）

生成時チェック:
  □ トレンドライン or 前期比較が含まれているか（RA-06 防止）
  □ 推奨アクションが含まれているか（RA-07 防止）
  □ 自動生成データに異常値がないか確認したか（RA-08 防止）
  □ Goodhart のゲーミング兆候がないか確認したか

配信時チェック:
  □ 適切な頻度か（RA-05 防止）
  □ 読者からフィードバックを受け取る仕組みがあるか
  □ 不要になったレポートを廃止する仕組みがあるか
```

## 6. Harvest との連携

```
Harvest での活用:
  1. Summary レポートで RA-01〜08 のアンチパターンチェックを適用
  2. Individual レポートで個人ランキングを含めない（RA-04 強制）
  3. 全レポートにトレンド（前期比）を標準搭載（RA-06 防止）
  4. レポートテンプレートに推奨アクションセクションを追加（RA-07 防止）
  5. ゲーミング検出パターンを Weekly レポートに統合

品質ゲート:
  - メトリクス数 > 10 で「情報過多」警告（RA-01 防止）
  - コンテキストなしの絶対値 → 前期比 or 目標比の追加を提案
  - 個人名 × 順位の組み合わせを検出 → 匿名化 or チーム集計を提案
  - レポートにアクション項目が 0 → 推奨アクションの追加を提案
```
