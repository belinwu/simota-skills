# Observability Anti-Patterns

> カーディナリティ爆発、メトリクスプロール、ダッシュボードスプロール、アラート疲れの検出と対策

## 1. 可観測性 6 大アンチパターン

| # | アンチパターン | 問題 | 影響 | 対策 |
|---|-------------|------|------|------|
| **OA-01** | **カーディナリティ爆発** | user_id / session_id / request_id 等の無制限ラベル | ストレージ指数膨張、クエリ遅延、コスト爆発 | 高カーディナリティラベル除去、ティア別保持ポリシー |
| **OA-02** | **メトリクススプロール** | 目的不明なメトリクス乱立 | ノイズ増大、シグナル埋没、運用コスト増 | メトリクスオーナー制度、定期棚卸し（四半期） |
| **OA-03** | **ダッシュボードスプロール** | チームごとに類似ダッシュボード乱造 | 情報分散、真実の単一源の喪失 | ダッシュボード階層設計（L0/L1/L2）、テンプレート化 |
| **OA-04** | **アラート疲れ** | 閾値ベースの過剰アラート | オオカミ少年効果、重要アラート見逃し | SLO ベースアラート、バーンレートアラートへ移行 |
| **OA-05** | **ツールスプロール** | Prometheus + Datadog + New Relic + 自前ツール並存 | コンテキストスイッチ増、コスト分散、相関困難 | OTel 統一パイプライン、単一バックエンド集約 |
| **OA-06** | **Collect-Everything 症候群** | 「念のため」全データ収集 | コスト膨張、S/N 比低下、セキュリティリスク | 目的駆動収集（SLO → SLI → 必要メトリクス逆算） |

## 2. カーディナリティ管理

```
カーディナリティ爆発の典型例:
  http_requests_total{method, path, status, user_id, client_ip}
  → method(5) × path(100) × status(10) × user_id(100K) × client_ip(50K)
  → 2.5 兆の一意時系列 → システム崩壊

検出方法:
  1. インデックスサイズ急増（RAM/ディスク消費監視）
  2. remote_write のインジェスト遅延
  3. 集約クエリ（sum, avg）のレイテンシ悪化
  4. 可観測性プラットフォームコストの急上昇

制御戦略:
  Tier 1: サービス別カーディナリティ上限設定
    - ビジネスクリティカルメトリクス: 高閾値許容
    - インフラメトリクス: 厳格な制限

  Tier 2: ティア別保持ポリシー
    - 高解像度（生データ）: 24-48 時間
    - 中解像度（1min 集約）: 30 日
    - 低解像度（1hr 集約）: 13+ ヶ月

  Tier 3: 適応型ダウンサンプリング
    - エッジ（ローカル）で高忠実度データ保持
    - 中央集約時に選択的ダウンサンプリング
    - 手動 recording rules より自動化を優先

圧縮技術:
  - Delta エンコーディング + XOR 圧縮（Prometheus/Mimir）
  - 辞書圧縮によるインデックスフットプリント削減
  - 注意: 圧縮はストレージを節約するがクエリ速度は改善しない
```

## 3. アラート疲れ対策フレームワーク

| レベル | 従来（閾値ベース） | 推奨（SLO ベース） |
|--------|------------------|------------------|
| **Critical** | CPU > 90% | エラーバジェット 3 日以内に枯渇（高速バーン） |
| **Warning** | CPU > 75% | エラーバジェット 50% 消費 |
| **Info** | CPU > 60% | SLI 劣化トレンド検出 |

```
アラート削減チェックリスト:
  □ 全アラートに Runbook がリンクされているか
  □ 過去 30 日でアクション不要だったアラートを特定→削除/調整
  □ 閾値ベースアラートを SLO バーンレートアラートに移行済みか
  □ アラートの重複（同一症状への複数アラート）を排除したか
  □ 営業時間外アラートのうち翌営業日対応可能なものを分離したか
  □ アラートの「おおかみ少年率」を計測しているか（目標: < 5%）
```

## 4. ダッシュボード階層設計

```
推奨階層:
  L0: エグゼクティブ概要（1 画面）
    - ビジネス SLO 状態（Green/Yellow/Red）
    - エラーバジェット残量
    - 主要サービスの健全性
    → 対象: 経営層、VP、プロダクトマネージャー

  L1: サービス概要（サービスごと 1 画面）
    - RED メトリクス（Rate/Error/Duration）
    - SLI トレンド（7 日/30 日）
    - 依存サービスの状態
    → 対象: オンコールエンジニア、SRE

  L2: デバッグ詳細（問題調査時）
    - リソース使用率（USE メソッド）
    - 分散トレース統合
    - ログ相関
    → 対象: 担当エンジニア

アンチパターン:
  ❌ L2 レベルの詳細をオンコールダッシュボードに含める
  ❌ チームごとに独自フォーマットのダッシュボードを作成
  ❌ ダッシュボードに 20+ パネルを詰め込む
  ❌ 説明なしのグラフ（何が正常か不明）
```

## 5. Beacon との連携

```
Beacon での活用:
  1. MEASURE モードで OA-01〜06 のスクリーニング
  2. DESIGN モードでダッシュボード階層・アラート戦略設計
  3. SPECIFY モードでカーディナリティ制御仕様の策定
  4. 定期レビューでメトリクス棚卸し・アラート削減

品質ゲート:
  - 新メトリクス追加時にカーディナリティ推定必須（OA-01 防止）
  - Runbook なしアラート → 作成を要求（OA-04 防止）
  - ダッシュボード作成時に L0/L1/L2 レベル指定必須（OA-03 防止）
  - 四半期ごとに未使用メトリクス棚卸し（OA-02 防止）
  - 収集目的不明のメトリクス → SLO 逆算で正当化要求（OA-06 防止）
```
