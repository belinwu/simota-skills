# SLO & Error Budget Anti-Patterns

> SLO 設計の落とし穴、エラーバジェットポリシー、バーンレートアラート、組織アラインメント

## 1. SLO 設計 8 大アンチパターン

| # | アンチパターン | 問題 | 影響 | 対策 |
|---|-------------|------|------|------|
| **SA-01** | **100% ターゲット** | SLO = 100% に設定 | デプロイ不可、パッチ不可、スケール不可 | 99.9% 以下の現実的ターゲット設定 |
| **SA-02** | **過去実績をそのまま SLO に** | 過去 p95 = 200ms → SLO = 200ms | 英雄的努力の常態化、改善余地ゼロ | ユーザー体験要件から逆算して設定 |
| **SA-03** | **可用性のみ注目** | SLO = 可用性 99.9% のみ | レイテンシ劣化、データ鮮度問題を見逃す | 多次元 SLI（可用性 + レイテンシ + 正確性） |
| **SA-04** | **バジェット月末チェック** | エラーバジェットを月末にのみ確認 | 急速消費に気付くのが遅すぎる | バーンレートアラートでリアルタイム監視 |
| **SA-05** | **外部依存の無視** | 依存サービスの信頼性を考慮しない | 自チームでは制御不能な SLO 違反 | 依存チェーン分析、依存 SLA の加味 |
| **SA-06** | **トラフィックパターン無視** | ピーク/オフピークで同一 SLO | ピーク時にバジェット急速消費 | 時間帯別 / シーズン別 SLO の検討 |
| **SA-07** | **組織アラインメント欠如** | エンジニアのみが SLO を理解 | PM/経営層との優先度不一致 | SLO ≡ ビジネス指標として全社共有 |
| **SA-08** | **ポリシーなき SLO** | SLO は定義したがポリシー未策定 | 違反しても何も変わらない（歯のない SLO） | エラーバジェットポリシーの明文化 |

## 2. エラーバジェット計算と管理

```
基本計算:
  SLO = 99.9%（28 日間、10M リクエスト）
  エラーバジェット = 100% - 99.9% = 0.1%
  許容エラー数 = 10,000,000 × 0.001 = 10,000 リクエスト

バーンレート:
  バーンレート = 実際のエラー率 / 許容エラー率
  バーンレート 1.0 = ちょうど 28 日でバジェット枯渇
  バーンレート 14.4 = 2 日でバジェット枯渇（Critical）

マルチウィンドウバーンレートアラート:
  | 優先度 | 条件 | アクション |
  |--------|------|----------|
  | P1 (Page) | 28 日バジェットが 3 日以内に枯渇 | 即時対応、オンコール通知 |
  | P2 (Ticket) | 四半期バジェットの 2% が 6 時間で消費 | チケット作成、翌営業日対応 |
  | P3 (Review) | 月間バジェット 50% 消費 | 週次レビューで議論 |
```

## 3. エラーバジェットポリシー

```
ポリシーフレームワーク:

  Green（バジェット残量 > 50%）:
    - 通常の機能開発を継続
    - リスクのあるデプロイも許可
    - 実験的変更を奨励

  Yellow（バジェット残量 25-50%）:
    - チーム分析ミーティング実施
    - 低リスクデプロイのみ許可
    - 高リスク変更は一時停止
    - 信頼性改善タスクの優先度を上げる

  Red（バジェット残量 < 25%）:
    - 機能開発を凍結
    - 信頼性改善に全リソース集中
    - リリースフリーズ（緊急パッチ除く）
    - バジェット回復まで維持

  ポリシー運用の鍵:
    - フリーズ宣言の権限者を明確に指定
    - VP/Director レベルのエスカレーション先
    - ポリシーが「罰」に感じる → SLO が厳しすぎるシグナル
    - サービス劣化がフリーズ前に発生 → SLO が緩すぎるシグナル
```

## 4. SLO ガバナンス

```
四半期レビューの必須項目:
  1. SLO 精度: ターゲットは適切か（厳しすぎ/緩すぎ）
  2. アラート有効性: バーンレートアラートは適時発火したか
  3. バジェット配分: どの変更がバジェットを消費したか
  4. ターゲット調整: 次四半期の SLO 変更が必要か

SLO 成熟度モデル:
  Level 1: SLI 定義済み、SLO 未設定
  Level 2: SLO 設定済み、手動モニタリング
  Level 3: バーンレートアラート運用、エラーバジェットポリシー策定
  Level 4: SLO ドリブン開発（バジェット消費に基づく優先度決定）
  Level 5: SLO 自動調整（トラフィックパターン・季節性に適応）

組織アラインメント:
  - 開発者: 自分のコードが SLO にどう影響するか理解
  - PM: 機能 vs 信頼性のトレードオフを理解
  - 経営層: エラーバジェット = イノベーション投資として認識
  - SRE: SLO の番人、ポリシー執行者
```

## 5. Beacon との連携

```
Beacon での活用:
  1. MEASURE モードで SA-01〜08 のスクリーニング
  2. SLO 設計時に多次元 SLI の策定（SA-03 防止）
  3. バーンレートアラート設計（SA-04 防止）
  4. 四半期レビューで SLO ガバナンス実施

品質ゲート:
  - SLO = 100% → 現実的ターゲットへの修正を要求（SA-01 防止）
  - 可用性のみの SLO → レイテンシ SLI 追加を推奨（SA-03 防止）
  - エラーバジェットポリシー未策定 → 策定を必須化（SA-08 防止）
  - バーンレートアラート未設定 → マルチウィンドウアラート設計を要求（SA-04 防止）
  - SLO レビュー未実施（> 1 四半期）→ レビュー実施を促進
```
